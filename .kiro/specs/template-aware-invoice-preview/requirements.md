# Requirements Document

## Introduction

The in-browser invoice preview in `InvoiceList.tsx` currently renders a hardcoded "default" style regardless of which PDF template the organisation has configured. The actual PDF (generated by WeasyPrint) correctly uses the org's selected template, but the HTML preview does not reflect this — creating a visual mismatch between what users see on-screen and what gets printed/emailed.

This feature updates the existing HTML preview component to apply template-specific CSS styling (colours, layout adjustments, font weights) based on the org's `invoice_template_id` from settings. The approach modifies the existing JSX structure's styling — it does NOT replace the HTML preview with a PDF embed, iframe, or object element.

## Glossary

- **Invoice_Preview**: The ~450-line JSX block in `InvoiceList.tsx` (lines 1253–1710) that renders an in-browser HTML representation of the selected invoice, including header, bill-to, line items, totals, payment history, notes, and conditional banners.
- **Template_Style_Map**: A frontend TypeScript module that maps each `invoice_template_id` to a set of CSS class names and inline style overrides that can be applied to the existing Invoice_Preview JSX structure.
- **Template_Registry**: The existing Python module (`app/modules/invoices/template_registry.py`) that defines all available invoice PDF templates and their metadata (colours, layout type, logo position).
- **Org_Settings**: The organisation's settings JSONB field which already contains `invoice_template_id` and `invoice_template_colours` (persisted by the existing Invoice Template settings tab).
- **Invoice_Detail_Response**: The JSON object returned by `GET /invoices/{id}` which currently includes org branding (name, logo, address) but does not include `invoice_template_id` or `invoice_template_colours`.
- **Primary_Colour**: The main brand colour used for the table header background, balance due bar, and key accent elements in the Invoice_Preview.
- **Accent_Colour**: A secondary colour used for hover states, borders, and secondary highlights in the Invoice_Preview.
- **Header_BG_Colour**: The background colour applied to the invoice header area in the Invoice_Preview.
- **Layout_Type**: Either `standard` or `compact` — compact templates use reduced padding on line-item rows and tighter spacing.
- **Logo_Position**: Either `left`, `center`, or `side` — determines the alignment of the org logo/name in the Invoice_Preview header.

## Requirements

### Requirement 1: Expose Template Settings in Invoice Detail Response

**User Story:** As a frontend developer, I want the invoice detail API to include the org's template settings, so that the preview component can apply the correct styling without an additional API call.

#### Acceptance Criteria

1. WHEN `GET /invoices/{id}` is called, THE Invoice_Detail_Response SHALL include the field `invoice_template_id` containing the org's configured template ID from Org_Settings, or `null` if no template is configured.
2. WHEN `GET /invoices/{id}` is called, THE Invoice_Detail_Response SHALL include the field `invoice_template_colours` containing the org's configured colour overrides object from Org_Settings, or `null` if no colour overrides are configured.
3. THE Invoice_Detail_Response SHALL include these fields by reading from the existing `org.settings` JSONB — no new database queries or joins SHALL be added.

### Requirement 2: Template Style Map Module

**User Story:** As a frontend developer, I want a centralised mapping of template IDs to CSS styles, so that the preview component can look up the correct styling for any template.

#### Acceptance Criteria

1. THE Template_Style_Map SHALL be defined in a new file `frontend/src/utils/invoiceTemplateStyles.ts`.
2. THE Template_Style_Map SHALL contain an entry for each template ID in the Template_Registry: `default`, `classic`, `modern-dark`, `bold-header`, `compact-blue`, `compact-green`, `compact-mono`, `corporate`, `elegant`, `minimal`, `ocean`, `sunrise`, `trade-pro`.
3. FOR EACH template entry, THE Template_Style_Map SHALL define: Primary_Colour, Accent_Colour, Header_BG_Colour, Logo_Position, and Layout_Type.
4. THE Template_Style_Map SHALL export a function that accepts a template ID and optional colour overrides, and returns a resolved style object with colour overrides taking precedence over template defaults.
5. WHEN the template ID is `null`, empty, or not found in the map, THE Template_Style_Map SHALL return the `default` template styles as a fallback.

### Requirement 3: Apply Template Colours to Invoice Preview

**User Story:** As an org admin, I want the in-browser invoice preview to use my configured template's colour scheme, so that the preview matches the actual PDF output.

#### Acceptance Criteria

1. WHEN an invoice is displayed in the Invoice_Preview, THE Invoice_Preview SHALL read `invoice_template_id` and `invoice_template_colours` from the Invoice_Detail_Response.
2. THE Invoice_Preview SHALL apply the resolved Primary_Colour to the line-items table header background and the balance due bar background.
3. THE Invoice_Preview SHALL apply the resolved Header_BG_Colour to the invoice header section background.
4. THE Invoice_Preview SHALL apply the resolved Accent_Colour to the "Bill To" section border and label colour.
5. WHEN `invoice_template_colours` contains overrides, THE Invoice_Preview SHALL use the override values instead of the template's default colours.
6. WHEN no template is configured (template ID is `null`), THE Invoice_Preview SHALL render with the current default styling (blue `#3b5bdb` header, white background) — preserving backward compatibility.

### Requirement 4: Apply Template Layout to Invoice Preview

**User Story:** As an org admin, I want the in-browser preview to reflect my template's layout style (compact vs standard, logo position), so that the preview accurately represents the PDF layout.

#### Acceptance Criteria

1. WHEN the resolved Layout_Type is `compact`, THE Invoice_Preview SHALL apply reduced vertical padding (no more than 6px top and bottom) to line-item table rows.
2. WHEN the resolved Layout_Type is `compact`, THE Invoice_Preview SHALL apply reduced overall padding to the invoice card sections.
3. WHEN the resolved Logo_Position is `center`, THE Invoice_Preview SHALL center-align the org logo/name in the header and move the invoice title to a separate row below.
4. WHEN the resolved Logo_Position is `side`, THE Invoice_Preview SHALL position the org logo/name on the right side of the header, with the invoice title and balance on the left.
5. WHEN the resolved Logo_Position is `left`, THE Invoice_Preview SHALL maintain the current layout (org info left, invoice title right) — this is the existing default behaviour.

### Requirement 5: Preserve All Existing Functionality

**User Story:** As a user, I want all existing invoice preview features to continue working after template styling is applied, so that no functionality is lost.

#### Acceptance Criteria

1. THE Invoice_Preview SHALL preserve the existing print functionality (Ctrl+P) with all print-specific CSS classes (`print-table-header`, `print-balance-bar`) intact and functional.
2. THE Invoice_Preview SHALL preserve the existing payment history section rendering, including all payment rows, refund notes, and summary totals.
3. THE Invoice_Preview SHALL preserve the existing internal notes section (amber background card) rendering and visibility.
4. THE Invoice_Preview SHALL preserve the existing POS receipt preview panel rendering alongside the invoice preview.
5. THE Invoice_Preview SHALL preserve all existing click handlers, including the invoice/receipt selection toggle and the credit note modal trigger.
6. THE Invoice_Preview SHALL preserve all existing conditional rendering: draft watermark, voided banner, status badges, vehicle info bars, additional vehicles, credit notes section, and empty-state placeholders.
7. THE Invoice_Preview SHALL NOT add any new API calls beyond reading the template fields already present in the Invoice_Detail_Response.
8. THE Invoice_Preview SHALL NOT add any new React state variables beyond what is needed to store the resolved template styles.
9. THE Invoice_Preview SHALL NOT replace the HTML JSX structure with a PDF embed, iframe, or object element.

### Requirement 6: Dark Header Template Readability

**User Story:** As an org admin using a dark-header template (e.g., modern-dark, bold-header), I want text in the header area to remain readable, so that the preview is usable.

#### Acceptance Criteria

1. WHEN the resolved Header_BG_Colour has a luminance below 0.5 (dark background), THE Invoice_Preview SHALL apply light text colour (white or near-white) to all text elements within the header section.
2. WHEN the resolved Header_BG_Colour has a luminance of 0.5 or above (light background), THE Invoice_Preview SHALL apply dark text colour (the default dark grays) to all text elements within the header section.
3. THE Template_Style_Map SHALL include a utility function that determines whether a given hex colour is "dark" (luminance < 0.5) to support this contrast logic.

### Requirement 7: Template Style Consistency with PDF

**User Story:** As an org admin, I want the in-browser preview colours to match the PDF output, so that there are no surprises when I download or email the invoice.

#### Acceptance Criteria

1. FOR EACH template in the Template_Style_Map, the default Primary_Colour, Accent_Colour, and Header_BG_Colour values SHALL match the corresponding values in the Template_Registry (`app/modules/invoices/template_registry.py`).
2. THE colour resolution logic in the Template_Style_Map (override > template default > fallback) SHALL follow the same precedence as the Template_Renderer in `generate_invoice_pdf()`.
3. THE Template_Style_Map SHALL be kept in sync with the Template_Registry — when a new template is added to the backend registry, a corresponding entry SHALL be added to the frontend style map.
