# Requirements Document

## Introduction

BudgetFlow requires a comprehensive Multi-Factor Authentication (MFA) system that allows users to protect their accounts using one or more second-factor methods: Email OTP, SMS OTP, TOTP (authenticator apps), and Passkeys (WebAuthn/FIDO2). Users can enable multiple methods simultaneously and generate single-use backup codes for account recovery. The system follows industry-standard practices (RFC 6238 for TOTP, 6-digit OTPs with expiry for email/SMS, WebAuthn Level 2 for passkeys) and integrates into the existing Profile/Security settings page.

## Glossary

- **MFA_System**: The Multi-Factor Authentication subsystem within BudgetFlow responsible for enrolling, verifying, managing, and enforcing second-factor authentication methods.
- **TOTP**: Time-based One-Time Password as defined by RFC 6238, generated by an authenticator app (e.g. Google Authenticator, Authy).
- **OTP**: A 6-digit one-time password sent via email or SMS with a limited validity window.
- **Backup_Code**: A single-use alphanumeric recovery code that can substitute for any MFA method during login.
- **MFA_Method**: One of four supported second-factor types: `totp`, `sms`, `email`, or `passkey`.
- **Passkey**: A FIDO2/WebAuthn credential bound to the user's device or security key, enabling passwordless or second-factor authentication via public-key cryptography.
- **WebAuthn**: The W3C Web Authentication API (Level 2) used by browsers and platforms to create and verify Passkey credentials.
- **Registration_Ceremony**: The WebAuthn process of creating a new Passkey credential, involving challenge generation, authenticator attestation, and public key storage.
- **Authentication_Ceremony**: The WebAuthn process of verifying a user's identity using a registered Passkey, involving challenge generation and authenticator assertion.
- **Relying_Party**: The BudgetFlow server that generates WebAuthn challenges and verifies authenticator responses, identified by its domain (RP ID).
- **Credential_ID**: A unique identifier assigned to a registered Passkey credential, used to reference and manage the credential.
- **Enrolment**: The process of setting up a new MFA method on a user account, including initial configuration and verification.
- **MFA_Settings_UI**: The frontend interface within the Profile/Settings page where users manage their MFA methods.
- **QR_Code**: A scannable image encoding the TOTP provisioning URI for authenticator app setup.
- **Provisioning_URI**: The `otpauth://` URI containing the TOTP secret, issuer, and account label.
- **Primary_Method**: The MFA method a user selects as their default for login challenges when multiple methods are enabled.

## Requirements

### Requirement 1: TOTP Enrolment

**User Story:** As a user, I want to set up TOTP-based MFA using my authenticator app, so that I can secure my account with a time-based code.

#### Acceptance Criteria

1. WHEN a user initiates TOTP enrolment, THE MFA_System SHALL generate a unique TOTP secret conforming to RFC 6238 with a 30-second time step and SHA-1 algorithm.
2. WHEN a TOTP secret is generated, THE MFA_System SHALL return a Provisioning_URI and a QR_Code image that encodes the Provisioning_URI with the issuer set to "BudgetFlow".
3. WHEN a user submits a valid 6-digit TOTP code during enrolment, THE MFA_System SHALL mark the TOTP method as verified and persist the encrypted secret to the user record.
4. WHEN a user submits an invalid TOTP code during enrolment, THE MFA_System SHALL reject the enrolment attempt and return a descriptive error message.
5. THE MFA_Settings_UI SHALL display the QR_Code and the TOTP secret in plain text as a manual entry fallback during enrolment.

### Requirement 2: SMS-Based MFA Enrolment

**User Story:** As a user, I want to set up SMS-based MFA using my phone number, so that I can receive verification codes via text message.

#### Acceptance Criteria

1. WHEN a user initiates SMS enrolment with a phone number, THE MFA_System SHALL send a 6-digit OTP to the provided phone number.
2. WHEN an SMS OTP is generated, THE MFA_System SHALL store the OTP with a validity window of 5 minutes.
3. WHEN a user submits the correct SMS OTP within the validity window, THE MFA_System SHALL mark the SMS method as verified and persist the phone number to the user record.
4. WHEN a user submits an incorrect or expired SMS OTP, THE MFA_System SHALL reject the verification and return a descriptive error message.
5. IF the SMS delivery fails, THEN THE MFA_System SHALL return an error indicating the SMS could not be delivered and allow the user to retry.
6. THE MFA_Settings_UI SHALL provide a phone number input field with international format validation during SMS enrolment.

### Requirement 3: Email-Based MFA Enrolment

**User Story:** As a user, I want to set up email-based MFA, so that I can receive verification codes at my registered email address.

#### Acceptance Criteria

1. WHEN a user initiates email MFA enrolment, THE MFA_System SHALL send a 6-digit OTP to the user's registered email address.
2. WHEN an email OTP is generated, THE MFA_System SHALL store the OTP with a validity window of 10 minutes.
3. WHEN a user submits the correct email OTP within the validity window, THE MFA_System SHALL mark the email method as verified on the user record.
4. WHEN a user submits an incorrect or expired email OTP, THE MFA_System SHALL reject the verification and return a descriptive error message.
5. IF the email delivery fails, THEN THE MFA_System SHALL return an error indicating the email could not be sent and allow the user to retry.

### Requirement 4: Multi-Method MFA Support

**User Story:** As a user, I want to enable multiple MFA methods simultaneously, so that I have flexible options for verifying my identity at login.

#### Acceptance Criteria

1. THE MFA_System SHALL allow a user to have up to four MFA methods enabled concurrently (one of each type: totp, sms, email, passkey), with multiple passkey credentials permitted under the single passkey method.
2. WHEN a user has multiple verified MFA methods, THE MFA_System SHALL present all enabled methods as options during the login MFA challenge.
3. WHEN a user selects an MFA method during login, THE MFA_System SHALL send or accept a code for the selected method only.
4. THE MFA_Settings_UI SHALL display the status (enabled/disabled) of each MFA method and allow independent enable/disable actions.
5. WHEN a user disables an MFA method, THE MFA_System SHALL remove the method from the user's verified methods list.
6. IF a user attempts to disable their last remaining MFA method and the organisation requires MFA, THEN THE MFA_System SHALL reject the request and inform the user that at least one method is required.

### Requirement 5: Backup Code Generation

**User Story:** As a user, I want to generate backup recovery codes, so that I can regain access to my account if I lose access to all my MFA methods.

#### Acceptance Criteria

1. WHEN a user requests backup code generation, THE MFA_System SHALL generate exactly 10 single-use alphanumeric Backup_Codes.
2. WHEN backup codes are generated, THE MFA_System SHALL store only the hashed values of the codes and return the plain-text codes to the user exactly once.
3. WHEN a user regenerates backup codes, THE MFA_System SHALL invalidate all previously generated backup codes before creating the new set.
4. THE MFA_Settings_UI SHALL display the generated backup codes with a clear warning that the codes are shown only once and cannot be retrieved again.
5. THE MFA_Settings_UI SHALL provide an option to copy all backup codes to the clipboard or download them as a text file.
6. WHEN a user uses a Backup_Code during login, THE MFA_System SHALL mark that specific code as consumed and reject subsequent uses of the same code.

### Requirement 6: MFA Login Challenge Flow

**User Story:** As a user with MFA enabled, I want to be prompted for my second factor after entering my password, so that my account is protected by two-factor verification.

#### Acceptance Criteria

1. WHEN a user with at least one verified MFA method provides valid credentials, THE MFA_System SHALL return an MFA challenge token instead of access tokens.
2. WHEN an MFA challenge is issued, THE MFA_System SHALL include the list of the user's enabled MFA method types in the challenge response.
3. WHEN the user selects the SMS method during an MFA challenge, THE MFA_System SHALL send a new 6-digit OTP to the user's registered phone number.
4. WHEN the user selects the email method during an MFA challenge, THE MFA_System SHALL send a new 6-digit OTP to the user's registered email address.
5. WHEN the user submits a valid code for the selected method, THE MFA_System SHALL issue access and refresh tokens.
6. IF a user fails MFA verification 5 consecutive times, THEN THE MFA_System SHALL lock the MFA challenge for that session and require the user to re-authenticate with credentials.
7. THE MFA_System SHALL expire the MFA challenge token after 5 minutes of inactivity.

### Requirement 7: MFA Method Removal and Disabling

**User Story:** As a user, I want to remove or disable an MFA method I no longer use, so that I can manage my security settings.

#### Acceptance Criteria

1. WHEN a user requests to disable an MFA method, THE MFA_System SHALL require the user to confirm their identity by providing their current password.
2. WHEN identity is confirmed, THE MFA_System SHALL remove the specified method from the user's verified MFA methods.
3. WHEN a TOTP method is removed, THE MFA_System SHALL delete the stored TOTP secret from the user record.
4. WHEN an SMS method is removed, THE MFA_System SHALL remove the associated phone number from the MFA configuration.
5. THE MFA_Settings_UI SHALL update the method status to disabled immediately after successful removal.

### Requirement 8: MFA Settings UI

**User Story:** As a user, I want a dedicated MFA management section in my profile settings, so that I can easily set up, view, and manage all my MFA methods.

#### Acceptance Criteria

1. THE MFA_Settings_UI SHALL replace the current read-only MFA display on the Profile page with an interactive management interface.
2. THE MFA_Settings_UI SHALL display each MFA method (TOTP, SMS, Email, Passkey) as a separate card with its current status (enabled or disabled).
3. WHEN a user clicks "Enable" on a disabled method card, THE MFA_Settings_UI SHALL open the enrolment flow for that specific method.
4. WHEN a user clicks "Disable" on an enabled method card, THE MFA_Settings_UI SHALL prompt for password confirmation before proceeding.
5. THE MFA_Settings_UI SHALL display a "Backup Codes" section showing whether backup codes have been generated and providing a button to generate or regenerate codes.
6. WHILE an enrolment flow is in progress, THE MFA_Settings_UI SHALL display a step-by-step guided setup with clear instructions for the selected method.
7. THE MFA_Settings_UI SHALL display a success confirmation after each method is successfully enrolled or disabled.

### Requirement 9: OTP Rate Limiting

**User Story:** As a system operator, I want OTP sending to be rate-limited, so that the system is protected from abuse and excessive SMS/email costs.

#### Acceptance Criteria

1. THE MFA_System SHALL limit OTP send requests to a maximum of 5 per method per user within a 15-minute window.
2. IF a user exceeds the OTP rate limit, THEN THE MFA_System SHALL reject the request with an error indicating the user must wait before requesting a new code.
3. THE MFA_System SHALL track OTP rate limits independently per MFA method type (SMS and email tracked separately).

### Requirement 10: MFA Enrolment Security

**User Story:** As a system operator, I want MFA enrolment to require authentication, so that only the account owner can modify MFA settings.

#### Acceptance Criteria

1. THE MFA_System SHALL require a valid authenticated session (JWT access token) for all MFA enrolment and management operations.
2. WHEN a user initiates any MFA enrolment, THE MFA_System SHALL verify the user's session is active and not expired.
3. THE MFA_System SHALL log all MFA enrolment, verification, and removal events for audit purposes.

### Requirement 11: Passkey Registration

**User Story:** As a user, I want to register a passkey (security key or platform authenticator) on my account, so that I can use it as a secure second-factor or passwordless login method.

#### Acceptance Criteria

1. WHEN a user initiates passkey registration, THE MFA_System SHALL generate a WebAuthn Registration_Ceremony challenge with the Relying_Party ID set to the BudgetFlow domain and a timeout of 60 seconds.
2. WHEN a user initiates passkey registration, THE MFA_System SHALL include all existing Credential_IDs for the user in the exclude list to prevent duplicate registrations of the same authenticator.
3. WHEN the browser returns a valid attestation response, THE MFA_System SHALL verify the attestation, extract the public key and Credential_ID, and persist the credential to the user record.
4. WHEN a user completes passkey registration, THE MFA_System SHALL store a user-provided friendly name (e.g. "Work Laptop", "YubiKey") alongside the credential for identification.
5. IF the attestation response is invalid or the Registration_Ceremony times out, THEN THE MFA_System SHALL reject the registration and return a descriptive error message.
6. THE MFA_System SHALL allow a user to register up to 10 passkey credentials on a single account.
7. THE MFA_Settings_UI SHALL display a guided passkey registration flow that invokes the browser WebAuthn API and prompts the user to interact with their authenticator.

### Requirement 12: Passkey Authentication

**User Story:** As a user with a registered passkey, I want to use my passkey during the MFA login challenge, so that I can verify my identity without typing a code.

#### Acceptance Criteria

1. WHEN a user selects the passkey method during an MFA challenge, THE MFA_System SHALL generate a WebAuthn Authentication_Ceremony challenge containing the user's registered Credential_IDs with a timeout of 60 seconds.
2. WHEN the browser returns a valid assertion response, THE MFA_System SHALL verify the signature against the stored public key and issue access and refresh tokens.
3. WHEN the assertion is verified, THE MFA_System SHALL update the sign count for the used credential and compare the new sign count against the stored value to detect cloned authenticators.
4. IF the assertion response is invalid, the signature verification fails, or the Authentication_Ceremony times out, THEN THE MFA_System SHALL reject the authentication attempt and return a descriptive error message.
5. IF the sign count from the authenticator is less than or equal to the stored sign count (indicating a possible cloned authenticator), THEN THE MFA_System SHALL reject the authentication and flag the credential for review.

### Requirement 13: Passkey Management

**User Story:** As a user, I want to view, rename, and remove my registered passkeys, so that I can keep my security credentials organized and up to date.

#### Acceptance Criteria

1. THE MFA_Settings_UI SHALL display a list of all registered passkeys for the user, showing the friendly name, registration date, and last used date for each credential.
2. WHEN a user requests to rename a passkey, THE MFA_System SHALL update the friendly name for the specified Credential_ID.
3. WHEN a user requests to remove a passkey, THE MFA_System SHALL require the user to confirm their identity by providing their current password.
4. WHEN identity is confirmed for passkey removal, THE MFA_System SHALL delete the specified credential (public key and Credential_ID) from the user record.
5. IF a user attempts to remove their last passkey credential and passkey is their only enabled MFA method and the organisation requires MFA, THEN THE MFA_System SHALL reject the request and inform the user that at least one MFA method must remain active.
6. THE MFA_Settings_UI SHALL provide "Rename" and "Remove" actions for each registered passkey in the passkey list.

### Requirement 14: Passkey Browser Compatibility

**User Story:** As a user, I want to know whether my browser supports passkeys before attempting registration, so that I am not presented with a flow that will fail.

#### Acceptance Criteria

1. WHEN the MFA_Settings_UI loads, THE MFA_Settings_UI SHALL check for WebAuthn API availability in the browser using the PublicKeyCredential interface.
2. IF the browser does not support the WebAuthn API, THEN THE MFA_Settings_UI SHALL disable the passkey registration option and display a message indicating that the current browser does not support passkeys.
3. WHILE the browser supports WebAuthn, THE MFA_Settings_UI SHALL enable the passkey registration and authentication options.
