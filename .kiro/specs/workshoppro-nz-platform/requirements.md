# Requirements Document

## Introduction

WorkshopPro NZ is a New Zealand-focused SaaS invoicing and business management platform built for workshops, garages, and service stations. It operates on a multi-tenant subscription model with modular features, enabling each subscribing organisation to configure the platform to their own branding, services, and billing needs. The platform is structured across three tiers: Global Admin Layer, Organisation Layer, and End User Layer. It uses PostgreSQL as the sole database engine, FastAPI (Python) backend, React + Tailwind CSS frontend, Redis caching, Stripe payments, Brevo/SendGrid email, and Twilio SMS. The GUI follows an Apple-like design philosophy — intuitive, clean, responsive across all devices, and requiring no training manual. All notifications and reminders are configurable (not automatic by default — once enabled, automatic until disabled). The platform must comply with the New Zealand Privacy Act 2020, IRD GST requirements, and industry-standard security practices.

## Glossary

- **Platform**: The WorkshopPro NZ SaaS application as a whole, including all tiers and modules
- **Global_Admin**: The platform owner role with access to the Global Admin Console; manages all organisations, integrations, billing, and platform health
- **Org_Admin**: The organisation administrator role; manages a single organisation's branding, settings, users, and billing
- **Salesperson**: The end-user role within an organisation; creates customers, vehicles, invoices, and records payments
- **Organisation**: A single subscribing workshop, garage, or service station with its own isolated data space
- **Tenant**: Synonym for Organisation in the multi-tenant architecture context
- **Invoice_Module**: The module responsible for creating, managing, searching, and issuing invoices
- **Payment_Module**: The module responsible for processing cash and Stripe payments, refunds, and payment history
- **Auth_Module**: The module responsible for authentication, MFA, session management, and account security
- **Org_Module**: The module responsible for organisation provisioning, branding, settings, and user management
- **Customer_Module**: The module responsible for customer record creation, search, merge, and Privacy Act compliance
- **Vehicle_Module**: The module responsible for vehicle lookup, Carjam integration, manual entry, and vehicle profiles
- **Catalogue_Module**: The module responsible for the service and parts catalogue configuration
- **Storage_Module**: The module responsible for document storage, quota enforcement, and bulk export/archive
- **Notification_Module**: The module responsible for email and SMS sending, templates, and automated reminders
- **Subscription_Module**: The module responsible for subscription plans, billing lifecycle, trials, and plan changes
- **Reporting_Module**: The module responsible for org-level and global-level reports and analytics
- **Admin_Console**: The Global Admin Console module including error logging, integration configuration, and organisation management
- **Carjam_API**: The New Zealand vehicle data lookup API used to retrieve vehicle details by registration number
- **Global_Vehicle_DB**: The shared PostgreSQL table storing all Carjam lookup results accessible to all organisations
- **GST**: Goods and Services Tax as defined by New Zealand Inland Revenue Department (IRD), default rate 15%
- **IRD**: Inland Revenue Department, New Zealand's tax authority
- **MFA**: Multi-Factor Authentication — requiring two or more verification methods for login
- **TOTP**: Time-based One-Time Password, a 6-digit rotating code generated by authenticator apps
- **Passkey**: A FIDO2/WebAuthn credential enabling passwordless login via biometric or device PIN
- **JWT**: JSON Web Token used for session authentication with short-lived access tokens and longer-lived refresh tokens
- **Stripe_Connect**: Stripe's platform for enabling organisations to connect their own Stripe accounts for payment processing
- **Credit_Note**: A document issued against an existing invoice to record a partial or full refund or correction
- **Dunning**: The process of sending automated payment failure notifications and retry attempts
- **Privacy_Act_2020**: New Zealand Privacy Act 2020 governing the collection, use, and disclosure of personal information
- **WOF**: Warrant of Fitness, a mandatory vehicle safety inspection in New Zealand
- **COF**: Certificate of Fitness, a mandatory safety inspection for heavy vehicles in New Zealand
- **PWA**: Progressive Web App, a web application that can be installed on devices and work offline
- **Webhook**: An HTTP callback triggered by an event in a third-party system (e.g. Stripe payment confirmation)
- **Job_Card**: A work order document describing the work to be performed on a vehicle before invoicing
- **Quote_Module**: The module responsible for creating, managing, and converting quotes/estimates to invoices
- **Booking_Module**: The module responsible for appointment scheduling and calendar management
- **Inventory_Module**: The module responsible for parts stock tracking, reorder alerts, and supplier management
- **Customer_Portal**: A self-service web interface for customers to view invoices, make payments, and view vehicle history
- **Fleet_Account**: A commercial customer account with multiple vehicles managed under a single billing arrangement
- **Xero_Integration**: Integration with Xero accounting software for automatic invoice and payment synchronisation
- **MYOB_Integration**: Integration with MYOB accounting software for automatic invoice and payment synchronisation

## Requirements

### Requirement 1: User Authentication and Login

**User Story:** As a platform user, I want to securely log in using my preferred authentication method, so that I can access the platform quickly and safely.

#### Acceptance Criteria

1. WHEN a user submits valid email and password credentials, THE Auth_Module SHALL authenticate the user and issue a JWT session with a 15-minute access token and a 7-day refresh token
2. WHEN a user selects "Remember this device", THE Auth_Module SHALL extend the refresh token lifetime to 30 days
3. WHEN a user selects "Sign in with Google", THE Auth_Module SHALL authenticate the user via Google OAuth 2.0 and create or link the account to the matching email address
4. WHEN a user authenticates using a registered Passkey (WebAuthn), THE Auth_Module SHALL complete authentication without requiring a password or additional MFA prompt
5. WHEN a refresh token is used more than once (indicating potential theft), THE Auth_Module SHALL invalidate the entire session family and send an email alert to the user
6. THE Auth_Module SHALL enforce HTTPS with TLS 1.3 and HSTS headers on all authentication endpoints
7. WHEN a user logs in successfully, THE Auth_Module SHALL record the timestamp, IP address, device type, and browser in the audit log
8. WHEN a login attempt fails, THE Auth_Module SHALL record the failure with timestamp, IP address, and reason in the audit log

### Requirement 2: Multi-Factor Authentication (MFA)

**User Story:** As an Org_Admin, I want to configure MFA policies for my organisation, so that user accounts are protected with additional verification.

#### Acceptance Criteria

1. THE Auth_Module SHALL support TOTP (authenticator app), SMS-based OTP (via Twilio), and email-based OTP as MFA methods
2. WHEN a user enrols in TOTP MFA, THE Auth_Module SHALL display a QR code for scanning with any standard authenticator app and require a valid 6-digit code to confirm enrolment
3. WHEN a user enrols in SMS-based OTP, THE Auth_Module SHALL send a 6-digit code to the user's registered mobile number and require entry of that code to confirm enrolment
4. WHEN a user enrols in email-based OTP, THE Auth_Module SHALL send a 6-digit code to the user's registered email address and require entry of that code to confirm enrolment
5. THE Auth_Module SHALL allow users to enrol in multiple MFA methods simultaneously with a configurable fallback chain
6. WHEN MFA enrolment is completed, THE Auth_Module SHALL generate a set of 10 single-use backup recovery codes that the user can download or print
7. WHEN an Org_Admin sets MFA to mandatory for their organisation, THE Auth_Module SHALL redirect any user without MFA enrolment to the MFA setup screen on next login and block access until setup is complete
8. THE Auth_Module SHALL require MFA on every login for all Global_Admin accounts regardless of organisation MFA settings
9. WHEN a user authenticates via Passkey, THE Auth_Module SHALL treat the authentication as satisfying MFA requirements (possession plus biometric/PIN)

### Requirement 3: Account Security and Session Management

**User Story:** As a platform user, I want my account protected against brute force attacks and unauthorised access, so that my data remains secure.

#### Acceptance Criteria

1. WHEN 5 consecutive failed login attempts occur for a single account, THE Auth_Module SHALL temporarily lock the account for 15 minutes
2. WHEN 10 consecutive failed login attempts occur for a single account, THE Auth_Module SHALL lock the account and send a manual unlock email to the user
3. THE Auth_Module SHALL enforce a minimum password length of 12 characters with complexity rules
4. WHEN a user sets or changes a password, THE Auth_Module SHALL check the password against the HaveIBeenPwned API using k-anonymity and warn the user if the password appears in a known breach
5. WHEN a login occurs from a new country, new device, or unusual time, THE Auth_Module SHALL send an email alert to the user with a "This wasn't me" link that revokes the session and locks the account
6. THE Auth_Module SHALL limit concurrent sessions to a configurable maximum per organisation (default 5 per user)
7. WHEN a user views their profile settings, THE Auth_Module SHALL display all active sessions with device type, IP address, and last activity timestamp, and allow remote termination of any session
8. WHEN an Org_Admin locks a user account, THE Auth_Module SHALL immediately invalidate all active sessions for that user

### Requirement 4: Password Recovery

**User Story:** As a platform user, I want to recover my account when I forget my password, so that I can regain access without contacting support.

#### Acceptance Criteria

1. WHEN a user requests a password reset, THE Auth_Module SHALL send a secure password reset link to the user's registered email address
2. THE Auth_Module SHALL expire password reset links after 1 hour
3. WHEN a password reset link is used, THE Auth_Module SHALL invalidate all existing sessions for that user
4. IF a password reset is requested for a non-existent email address, THEN THE Auth_Module SHALL display the same confirmation message as for a valid email to prevent account enumeration
5. WHEN a user has MFA backup codes, THE Auth_Module SHALL allow account recovery using a valid backup code if all MFA methods are unavailable

### Requirement 5: Role-Based Access Control

**User Story:** As a platform operator, I want strict role-based permissions enforced at every level, so that users can only access functionality appropriate to their role.

#### Acceptance Criteria

1. THE Auth_Module SHALL enforce three roles: Global_Admin, Org_Admin, and Salesperson
2. THE Auth_Module SHALL verify role and organisation membership on every API request before processing
3. WHEN a user with the Salesperson role accesses the platform, THE Auth_Module SHALL hide all menu items and UI elements for organisation settings, billing, user management, and service catalogue configuration
4. WHEN a user with the Org_Admin role accesses the platform, THE Auth_Module SHALL grant access to all Salesperson functions plus organisation settings, billing, user management, and service catalogue configuration
5. WHEN a user with the Global_Admin role accesses the Global Admin Console, THE Auth_Module SHALL grant access to all platform management functions but restrict access to individual organisation customer and invoice data
6. THE Auth_Module SHALL enforce tenant isolation so that no API request from one organisation can access data belonging to another organisation

### Requirement 6: IP Allowlisting

**User Story:** As an Org_Admin, I want to restrict login access to specific IP ranges, so that only users on trusted networks can access the organisation.

#### Acceptance Criteria

1. WHERE IP allowlisting is enabled by the Org_Admin, THE Auth_Module SHALL reject login attempts from IP addresses outside the configured ranges
2. WHEN an Org_Admin configures IP allowlisting, THE Auth_Module SHALL validate that the current session IP is included in the allowlist before saving to prevent self-lockout
3. IF a login attempt is blocked by IP allowlisting, THEN THE Auth_Module SHALL return a clear error message indicating the IP is not authorised and log the attempt in the audit log

### Requirement 7: Email Verification

**User Story:** As a platform operator, I want all user email addresses verified at account creation, so that notifications and password resets reach the correct recipient.

#### Acceptance Criteria

1. WHEN a new user account is created via invitation, THE Auth_Module SHALL send a verification email with a secure signup link valid for 48 hours
2. WHEN a user clicks the verification link, THE Auth_Module SHALL mark the email as verified and allow the user to set their password and complete account setup
3. IF a verification link expires, THEN THE Auth_Module SHALL allow the Org_Admin to resend the invitation

### Requirement 8: Organisation Provisioning and Onboarding

**User Story:** As a Global_Admin, I want to provision new organisations and guide them through setup, so that workshops can start using the platform quickly.

#### Acceptance Criteria

1. WHEN a Global_Admin provisions a new organisation, THE Org_Module SHALL create the organisation record, assign a subscription plan, and generate an Org_Admin invitation email
2. WHEN an Org_Admin first logs in, THE Org_Module SHALL present a step-by-step onboarding wizard covering: organisation name and contact details, logo upload, brand colour selection, GST number entry, GST percentage (default 15%), invoice numbering prefix and starting number, default payment terms, and adding at least one service type
3. THE Org_Module SHALL display a visible progress indicator during the onboarding wizard with one screen per step
4. THE Org_Module SHALL allow any onboarding step to be skipped and completed later from Settings
5. THE Org_Module SHALL make the workspace usable immediately after the wizard regardless of completion state
6. WHEN a new workshop signs up via the public signup page, THE Subscription_Module SHALL create the organisation, start a 14-day free trial, collect credit card details via Stripe SetupIntent without charging, and trigger the onboarding wizard

### Requirement 9: Organisation Branding and Settings

**User Story:** As an Org_Admin, I want to configure my organisation's branding and business settings, so that invoices and communications reflect my workshop's identity.

#### Acceptance Criteria

1. THE Org_Module SHALL allow Org_Admin to configure: organisation name, logo (PNG or SVG), primary colour, secondary colour, address, phone number, email, invoice header text, invoice footer text, and custom email signature
2. THE Org_Module SHALL display the configured logo on all invoice PDFs and in the application header
3. THE Org_Module SHALL allow Org_Admin to configure GST number (validated against IRD format), GST percentage, and a toggle for GST-inclusive or GST-exclusive default display on invoices
4. THE Org_Module SHALL allow Org_Admin to configure invoice number prefix (e.g. "INV-", "WS-"), starting invoice number, default due date in days from issue date, and default invoice notes
5. THE Org_Module SHALL allow Org_Admin to configure payment terms as a number of days, a custom payment terms statement for invoice PDFs, and whether to allow partial payments
6. THE Org_Module SHALL allow Org_Admin to write custom terms and conditions using a rich text editor with headings, bold, bullet lists, and links, displayed at the bottom of every invoice PDF
7. THE Org_Module SHALL support multiple branch locations per organisation, each with its own address and phone number appearing on invoices for jobs done at that branch
8. THE Org_Module SHALL allow users to be assigned to one or more branches within the organisation

### Requirement 10: Organisation User Management

**User Story:** As an Org_Admin, I want to manage user accounts within my organisation, so that I can control who has access and what they can do.

#### Acceptance Criteria

1. WHEN an Org_Admin invites a new user, THE Org_Module SHALL send an email with a secure signup link valid for 48 hours and allow role assignment as Org_Admin or Salesperson
2. THE Org_Module SHALL allow Org_Admin to deactivate user accounts, which immediately invalidates all active sessions for that user
3. THE Org_Module SHALL allow Org_Admin to configure whether MFA is optional or mandatory for all users in the organisation
4. THE Org_Module SHALL enforce user seat limits based on the organisation's subscription plan
5. WHEN the user seat limit is reached, THE Org_Module SHALL prevent new user invitations and display a message directing the Org_Admin to upgrade their plan

### Requirement 11: Customer Creation and Search

**User Story:** As a Salesperson, I want to quickly find or create customer records during invoice creation, so that I can serve customers without delays.

#### Acceptance Criteria

1. WHEN a Salesperson types into the customer search field, THE Customer_Module SHALL perform a live search against customer name, phone number, and email simultaneously and display results as a dropdown
2. THE Customer_Module SHALL display each search result with customer name, phone, and email for quick identification
3. WHEN no matching customer is found, THE Customer_Module SHALL display a "Create new customer" option directly in the search dropdown
4. WHEN the user selects "Create new customer", THE Customer_Module SHALL open a minimal inline form (first name, last name, email, phone, optional address) without leaving the invoice creation flow
5. THE Customer_Module SHALL store customer records with: first name, last name, email address, phone number, optional physical address, and an internal notes field
6. THE Customer_Module SHALL enforce that customer records exist only within their organisation and are never shared across organisations

### Requirement 12: Customer Profile and History

**User Story:** As an Org_Admin or Salesperson, I want to view a complete customer profile with all linked vehicles and invoice history, so that I can understand the customer's relationship with the workshop.

#### Acceptance Criteria

1. THE Customer_Module SHALL display on the customer profile page: all linked vehicles, full invoice history across all vehicles, total spend, and outstanding balance
2. THE Customer_Module SHALL allow sending a one-off email or SMS directly from the customer profile
3. THE Customer_Module SHALL allow tagging a vehicle to a customer during creation or at any time from the customer profile
4. THE Customer_Module SHALL support customer record merging when duplicates are created, combining invoice history, vehicles, and contact details into a single record with a confirmation screen showing exactly what will be combined

### Requirement 13: Customer Data Privacy (Privacy Act 2020)

**User Story:** As an Org_Admin, I want to fulfil customer data access and deletion requests, so that the organisation complies with the New Zealand Privacy Act 2020.

#### Acceptance Criteria

1. WHEN a customer requests deletion of their record, THE Customer_Module SHALL allow the Org_Admin to process the request from the customer profile
2. WHEN a customer record is deleted, THE Customer_Module SHALL anonymise linked invoice records (replacing customer name with "Anonymised Customer" and clearing contact details) rather than deleting invoices, so financial records remain intact
3. WHEN a customer requests access to their data, THE Customer_Module SHALL allow the Org_Admin to export all data for that customer in JSON format
4. THE Customer_Module SHALL never write personally identifiable information (names, emails, phone numbers, addresses) to application logs or error reports

### Requirement 14: Vehicle Lookup via Carjam Integration

**User Story:** As a Salesperson, I want vehicle details to auto-populate when I type a registration number, so that I can create invoices quickly without manual data entry.

#### Acceptance Criteria

1. WHEN a user types a registration number, THE Vehicle_Module SHALL first check the Global_Vehicle_DB for a matching record
2. WHEN a match is found in the Global_Vehicle_DB, THE Vehicle_Module SHALL return the vehicle data instantly without making a Carjam_API call and without incrementing the organisation's Carjam usage counter
3. WHEN no match is found in the Global_Vehicle_DB, THE Vehicle_Module SHALL call the Carjam_API, store the result in the Global_Vehicle_DB, and increment the organisation's Carjam usage counter by one
4. THE Vehicle_Module SHALL store the following fields from Carjam: rego, make, model, year, colour, body type, fuel type, engine size, number of seats, current WOF expiry date, current registration expiry date, odometer at last recorded point, and the timestamp of when data was last pulled
5. WHEN a user clicks the "Refresh" button on a vehicle record, THE Vehicle_Module SHALL force a new Carjam_API call, update the Global_Vehicle_DB record, and charge the organisation for one lookup
6. IF the Carjam_API returns no result for a registration number, THEN THE Vehicle_Module SHALL present a manual entry form for make, model, year, colour, and other known details
7. WHEN a vehicle is manually entered, THE Vehicle_Module SHALL store the record in the organisation's own vehicle records (not the Global_Vehicle_DB) and mark it as "manually entered"

### Requirement 15: Vehicle Linking and Profile

**User Story:** As a Salesperson, I want to view a vehicle's complete service history and link vehicles to customers, so that I can provide informed service.

#### Acceptance Criteria

1. THE Vehicle_Module SHALL allow the same Global_Vehicle_DB record to be linked to different customers across different organisations without conflict
2. THE Vehicle_Module SHALL allow a vehicle to be linked to multiple customers within a single organisation
3. THE Vehicle_Module SHALL display on the vehicle profile page: all Carjam data, linked customer(s), odometer at each service visit, full service history (all invoices for that rego in chronological order), and upcoming WOF and registration expiry dates
4. THE Vehicle_Module SHALL display WOF and registration expiry dates with a visual indicator: green for more than 60 days, amber for 30–60 days, red for under 30 days or expired

### Requirement 16: Carjam Usage Monitoring and Billing

**User Story:** As a Global_Admin, I want to monitor Carjam API usage per organisation and bill for overages, so that API costs are recovered.

#### Acceptance Criteria

1. THE Admin_Console SHALL display a real-time table of Carjam API usage per organisation: total lookups this month, lookups included in plan, overage lookups, and overage charge accrued
2. THE Vehicle_Module SHALL enforce a global rate limit (configurable by Global_Admin) on maximum Carjam API calls per minute across the entire platform
3. WHEN an organisation exceeds their plan's included Carjam lookups, THE Subscription_Module SHALL automatically add the overage charge to the organisation's next monthly Stripe invoice
4. THE Org_Module SHALL display the organisation's Carjam usage on the billing dashboard so Org_Admins are aware of accrued charges

### Requirement 17: Invoice Creation

**User Story:** As a Salesperson, I want to create invoices in a fast, single-screen flow, so that I can process jobs efficiently at the service counter.

#### Acceptance Criteria

1. THE Invoice_Module SHALL present invoice creation as a single-screen form with a logical top-to-bottom flow: customer selection, vehicle selection, line items, totals, and payment/save actions
2. THE Invoice_Module SHALL render the form as a two-column layout on desktop and a single-column layout with large touch-friendly inputs on mobile
3. WHEN a customer is selected or created, THE Invoice_Module SHALL link the customer to the invoice immediately
4. WHEN a registration number is entered, THE Invoice_Module SHALL auto-populate all vehicle fields from the Vehicle_Module
5. THE Invoice_Module SHALL auto-calculate subtotal (ex-GST), GST amount, and GST-inclusive total as line items are added or modified
6. THE Invoice_Module SHALL allow saving an invoice as Draft (not yet issued, no invoice number assigned, editable) or issuing it (assigns sequential invoice number with org prefix, locks structural edits)

### Requirement 18: Invoice Line Items

**User Story:** As a Salesperson, I want to add services, parts, and labour to an invoice with flexible pricing, so that I can accurately bill for all work performed.

#### Acceptance Criteria

1. THE Invoice_Module SHALL support three line item types: Service, Part, and Labour
2. WHEN a Service line item is added, THE Invoice_Module SHALL allow selection from the organisation's service catalogue with the catalogue price pre-filled but fully editable
3. WHEN a Part line item is added, THE Invoice_Module SHALL provide fields for description, optional part number, quantity, and unit price
4. WHEN a Labour line item is added, THE Invoice_Module SHALL provide fields for description, hours, and hourly rate (selectable from configured labour rates)
5. THE Invoice_Module SHALL allow a free-text warranty note to be attached to any line item, displayed under that line on the invoice PDF
6. THE Invoice_Module SHALL allow individual line items to be marked as GST-exempt
7. THE Invoice_Module SHALL support discounts per line item (percentage or fixed amount) and invoice-level discounts applied to the subtotal

### Requirement 19: Invoice Statuses and Lifecycle

**User Story:** As a Salesperson, I want clear invoice statuses that update automatically, so that I always know the payment state of every invoice.

#### Acceptance Criteria

1. THE Invoice_Module SHALL support the following statuses: Draft, Issued, Partially Paid, Paid, Overdue, and Voided
2. WHILE an invoice is in Draft status, THE Invoice_Module SHALL allow free editing of all fields and not assign an invoice number
3. WHEN an invoice is issued, THE Invoice_Module SHALL assign a sequential invoice number with the organisation's configured prefix and lock structural edits (items, pricing) while allowing notes updates
4. WHEN one or more payments are recorded but the balance is not cleared, THE Invoice_Module SHALL set the status to Partially Paid
5. WHEN the full balance is cleared, THE Invoice_Module SHALL set the status to Paid
6. WHEN an issued invoice's due date passes with an outstanding balance, THE Invoice_Module SHALL automatically update the status to Overdue at midnight on the due date
7. WHEN an invoice is voided, THE Invoice_Module SHALL retain the invoice number in sequence, mark it as void, require a reason for voiding (recorded in the audit log), and exclude it from revenue reporting

### Requirement 20: Credit Notes

**User Story:** As an Org_Admin, I want to issue credit notes against invoices, so that corrections and refunds are properly documented for accounting compliance.

#### Acceptance Criteria

1. WHEN a Paid or Issued invoice needs correction or refund, THE Invoice_Module SHALL create a Credit_Note as a separate document linked to the original invoice with its own reference number (e.g. CN-0001 linked to INV-0047)
2. THE Invoice_Module SHALL display on the Credit_Note what is being credited and the reason
3. WHEN a Credit_Note is issued, THE Invoice_Module SHALL automatically update the net balance on the original invoice and adjust revenue reporting
4. WHEN a Credit_Note is issued against an invoice with a Stripe payment, THE Invoice_Module SHALL prompt the user to process a refund through Stripe for the credited amount

### Requirement 21: Invoice Search and Filtering

**User Story:** As a Salesperson, I want to search invoices by multiple criteria, so that I can quickly find any invoice.

#### Acceptance Criteria

1. THE Invoice_Module SHALL allow searching across: invoice number, registration number, customer name, customer phone, customer email, and date range
2. THE Invoice_Module SHALL display search results instantly as the user types
3. THE Invoice_Module SHALL allow filters to be stacked (e.g. all overdue invoices for a specific rego)
4. THE Invoice_Module SHALL display search results showing invoice number, customer name, rego, total, status, and issue date in a scannable list

### Requirement 22: Invoice Duplication

**User Story:** As a Salesperson, I want to duplicate an existing invoice, so that I can quickly create invoices for repeat services.

#### Acceptance Criteria

1. WHEN a user clicks "Duplicate" on any invoice, THE Invoice_Module SHALL create a new Draft pre-filled with the same customer, vehicle, and line items
2. THE Invoice_Module SHALL not assign an invoice number to the duplicated draft until it is issued

### Requirement 23: Invoice Number Integrity

**User Story:** As a platform operator, I want invoice numbers to be tamper-evident and contiguous, so that the system meets accounting compliance requirements.

#### Acceptance Criteria

1. THE Invoice_Module SHALL generate invoice numbers in a contiguous, sequential order per organisation with no gaps (excluding voided invoices which retain their number)
2. THE Invoice_Module SHALL prevent any modification to assigned invoice numbers through the application or API
3. THE Invoice_Module SHALL record all invoice state changes (creation, issuance, payment, voiding) in the tamper-evident audit log with before and after values

### Requirement 24: Cash Payment Processing

**User Story:** As a Salesperson, I want to record cash payments against invoices, so that I can update payment status immediately.

#### Acceptance Criteria

1. WHEN a Salesperson records a cash payment, THE Payment_Module SHALL capture the amount received, record the payment with a timestamp and the user who recorded it
2. WHEN the cash payment amount clears the invoice balance, THE Payment_Module SHALL update the invoice status to Paid
3. WHEN the cash payment amount is less than the outstanding balance, THE Payment_Module SHALL update the invoice status to Partially Paid and display the remaining balance

### Requirement 25: Stripe Payment Processing

**User Story:** As an Org_Admin, I want to connect my organisation's Stripe account and accept card payments, so that customers can pay invoices electronically.

#### Acceptance Criteria

1. THE Payment_Module SHALL use Stripe Connect for organisation-level payment setup, with a single "Connect Stripe Account" button that redirects the Org_Admin through Stripe's secure OAuth flow
2. THE Payment_Module SHALL never handle raw card data — all card input is handled by Stripe.js hosted fields
3. WHEN an invoice is ready for card payment, THE Payment_Module SHALL allow the Salesperson to generate a secure payment link sent to the customer via email or SMS
4. WHEN a Stripe payment is confirmed via webhook, THE Payment_Module SHALL update the invoice status in real time and automatically email a payment receipt to the customer
5. THE Payment_Module SHALL support partial Stripe payments for deposit scenarios

### Requirement 26: Payment History and Refunds

**User Story:** As an Org_Admin, I want a complete payment history on every invoice and the ability to process refunds, so that financial records are accurate and auditable.

#### Acceptance Criteria

1. THE Payment_Module SHALL display on every invoice a full payment history: each payment with date, amount, method (Cash or Stripe), and the user who recorded it
2. WHEN a refund is initiated for a Stripe payment, THE Payment_Module SHALL process the refund via the Stripe API and update the invoice balance
3. WHEN a refund is recorded for a cash payment, THE Payment_Module SHALL record the refund manually with a note and update the invoice balance
4. THE Payment_Module SHALL include all payment and refund events in the audit log

### Requirement 27: Service Catalogue Configuration

**User Story:** As an Org_Admin, I want to configure a catalogue of services with default pricing, so that Salespeople can quickly add standard services to invoices.

#### Acceptance Criteria

1. THE Catalogue_Module SHALL allow Org_Admin to create service entries with: service name, description, default price (ex-GST), GST applicability toggle, service category (e.g. Warrant, Service, Repair, Diagnostic), and an active/inactive toggle
2. WHEN a service is set to inactive, THE Catalogue_Module SHALL hide it from the invoice creation form but retain it for historical invoice display
3. THE Catalogue_Module SHALL allow catalogue prices to be overridden per invoice line item without affecting the catalogue default

### Requirement 28: Parts Catalogue and Labour Rates

**User Story:** As an Org_Admin, I want to optionally pre-load common parts and configure labour rates, so that invoice creation is faster and more consistent.

#### Acceptance Criteria

1. THE Catalogue_Module SHALL allow Org_Admin to pre-load parts with: part name, part number, default price, and supplier (optional)
2. THE Catalogue_Module SHALL allow parts to be added ad-hoc per invoice without pre-loading in the catalogue
3. THE Catalogue_Module SHALL allow Org_Admin to configure labour rates (e.g. standard rate, specialist rate) with a name and hourly rate that can be selected when adding Labour line items

### Requirement 29: Storage Quota Enforcement

**User Story:** As a platform operator, I want storage quotas enforced per organisation with clear warnings, so that platform resources are managed fairly.

#### Acceptance Criteria

1. THE Storage_Module SHALL calculate storage usage from the total size of all compressed invoice JSON records, customer records, and vehicle records per organisation (logos and branding assets excluded)
2. WHEN an organisation reaches 80% of their storage quota, THE Storage_Module SHALL display a persistent amber banner at the top of the dashboard for Org_Admin users and send an email notification to all Org_Admins
3. WHEN an organisation reaches 90% of their storage quota, THE Storage_Module SHALL display a prominent red alert visible to all users in the organisation, send an urgent email to all Org_Admins, and show current usage, limit, estimated remaining invoices, and a button to purchase additional storage
4. WHEN an organisation reaches 100% of their storage quota, THE Storage_Module SHALL block invoice creation, display a full-page interstitial to Salespeople directing them to contact their Org_Admin, and display the same interstitial to Org_Admins with the option to purchase more storage immediately
5. WHILE an organisation is at 100% storage, THE Storage_Module SHALL allow all other functionality (viewing, searching, payments) to continue normally

### Requirement 30: Storage Add-On Purchasing

**User Story:** As an Org_Admin, I want to purchase additional storage instantly, so that my team can continue creating invoices without waiting.

#### Acceptance Criteria

1. THE Storage_Module SHALL allow Org_Admin to purchase storage add-ons from the Billing page in increments configured by Global_Admin (e.g. +5GB, +20GB, +50GB)

4. THE Platform SHALL enforce a separate, stricter rate limit on authentication endpoints (default: 10 requests per minute per IP) to prevent brute force attacks

### Requirement 72: Search and Filtering Across All Modules

**User Story:** As a platform user, I want fast, consistent search and filtering across all modules, so that I can find any record quickly.

#### Acceptance Criteria

1. THE Platform SHALL provide a global search bar accessible from any screen that searches across customers, vehicles (by rego), and invoices simultaneously
2. THE Platform SHALL display search results grouped by type (Customer, Vehicle, Invoice) with the most relevant results first
3. THE Platform SHALL return search results within 500 milliseconds for datasets up to 100,000 records per organisation
4. THE Platform SHALL support filtering on all list views with stackable filters relevant to the data type (e.g. status, date range, amount range)

### Requirement 73: Dashboard and Home Screen

**User Story:** As a platform user, I want a role-appropriate dashboard when I log in, so that I immediately see the most important information for my work.

#### Acceptance Criteria

1. WHEN a Salesperson logs in, THE Platform SHALL display a dashboard showing: today's appointments (if booking module enabled), active Job Cards, recent invoices, overdue invoices requiring attention, and a quick-action button to create a new invoice
2. WHEN an Org_Admin logs in, THE Platform SHALL display a dashboard showing: revenue summary for the current period, outstanding invoice total, overdue invoice count, storage usage, recent activity feed, and any system alerts (storage warnings, billing issues)
3. WHEN a Global_Admin logs in, THE Admin_Console SHALL display a dashboard showing: platform MRR, total active organisations, recent error count by severity, integration health status, and any organisations with billing issues

### Requirement 74: Keyboard Shortcuts

**User Story:** As a power user, I want keyboard shortcuts for common actions, so that I can work faster without reaching for the mouse.

#### Acceptance Criteria

1. THE Platform SHALL support keyboard shortcuts for: new invoice (Ctrl/Cmd+N), search (Ctrl/Cmd+K), save draft (Ctrl/Cmd+S), and navigation between modules
2. THE Platform SHALL display a keyboard shortcut reference accessible via a help menu or shortcut (Ctrl/Cmd+/)
3. THE Platform SHALL not conflict keyboard shortcuts with browser-native shortcuts

### Requirement 75: Print-Friendly Views

**User Story:** As a Salesperson, I want to print invoices and reports directly from the browser, so that I can provide paper copies when needed.

#### Acceptance Criteria

1. THE Platform SHALL provide print-optimised CSS for invoice detail views, customer statements, and all reports
2. WHEN a user triggers browser print on an invoice or report, THE Platform SHALL render a clean, branded layout without navigation elements, sidebars, or interactive controls
3. THE Platform SHALL provide a "Print" button on invoice detail and report pages that triggers the browser print dialog

### Requirement 76: Progressive Web App (PWA) Support

**User Story:** As a workshop user, I want to install the platform as an app on my device, so that I can access it quickly from my home screen.

#### Acceptance Criteria

1. THE Platform SHALL provide a valid PWA manifest enabling installation on mobile and desktop devices
2. THE Platform SHALL cache critical application assets via a service worker for fast loading on repeat visits
3. THE Platform SHALL display an "Install App" prompt on supported devices when the user visits the platform repeatedly

### Requirement 77: Offline Capability for Workshop Floor

**User Story:** As a Salesperson, I want basic functionality available when the internet connection drops briefly, so that I can continue working on the workshop floor.

#### Acceptance Criteria

1. WHILE the device is offline, THE Platform SHALL allow the user to view previously loaded invoices, customers, and vehicle records from the local cache
2. WHILE the device is offline, THE Platform SHALL allow the user to start creating a new invoice and save it locally
3. WHEN the internet connection is restored, THE Platform SHALL automatically sync locally saved data to the server and notify the user of successful sync
4. IF a sync conflict occurs (e.g. the same record was modified on another device), THEN THE Platform SHALL present both versions to the user and allow them to choose which to keep

### Requirement 78: Batch Operations

**User Story:** As an Org_Admin, I want to perform bulk actions on invoices and customers, so that I can manage large volumes efficiently.

#### Acceptance Criteria

1. THE Platform SHALL allow Org_Admin to select multiple invoices and perform batch actions: mark as paid, send payment reminders, export as PDF ZIP, and export as CSV
2. THE Platform SHALL allow Org_Admin to bulk-import customers from a CSV file with validation and preview before committing
3. THE Platform SHALL display a progress indicator during batch operations and provide a summary of results (successful, failed, skipped) on completion

### Requirement 79: Multi-Currency Support

**User Story:** As an Org_Admin, I want to issue invoices in currencies other than NZD when needed, so that I can serve international customers or suppliers.

#### Acceptance Criteria

1. THE Invoice_Module SHALL default to NZD as the primary currency for all organisations
2. WHERE multi-currency is enabled by the Org_Admin, THE Invoice_Module SHALL allow selecting a currency per invoice from a configurable list of supported currencies
3. THE Invoice_Module SHALL display the currency symbol and code on the invoice PDF and in all totals
4. THE Reporting_Module SHALL report all revenue in NZD, converting foreign currency invoices at the exchange rate recorded at the time of invoice creation

### Requirement 80: Tax Invoice Compliance

**User Story:** As a platform operator, I want all invoices to meet NZ tax invoice requirements, so that organisations can use them for GST returns.

#### Acceptance Criteria

1. THE Invoice_Module SHALL include on every tax invoice: the words "Tax Invoice", the supplier's name and GST number, the invoice date, a description of the goods or services, the total amount including GST, and the GST amount
2. WHEN an invoice total exceeds $1,000 NZD (including GST), THE Invoice_Module SHALL also include the buyer's name and address on the invoice as required by IRD for high-value tax invoices
3. THE Invoice_Module SHALL clearly distinguish between taxable and GST-exempt line items on the invoice PDF

### Requirement 81: Performance Requirements

**User Story:** As a platform operator, I want the application to meet defined performance targets, so that users experience a fast, responsive interface.

#### Acceptance Criteria

1. THE Platform SHALL render any page within 2 seconds on a standard broadband connection (10 Mbps)
2. THE Platform SHALL respond to API requests within 200 milliseconds for standard CRUD operations under normal load
3. THE Platform SHALL support at least 500 concurrent users across all organisations without degradation
4. THE Platform SHALL use Redis caching for frequently accessed data (vehicle lookups, service catalogues, session data) to reduce database load
5. THE Platform SHALL use database connection pooling and query optimisation to maintain response times as data volume grows

### Requirement 82: Scalability

**User Story:** As a platform operator, I want the architecture to scale horizontally, so that the platform can grow to support thousands of organisations.

#### Acceptance Criteria

1. THE Platform SHALL use a stateless backend architecture (FastAPI) that supports horizontal scaling behind a load balancer
2. THE Platform SHALL use PostgreSQL row-level security for tenant isolation that scales without requiring separate database instances per organisation
3. THE Platform SHALL use asynchronous task processing (e.g. Celery with Redis) for background jobs (notifications, PDF generation, report generation) to prevent blocking the main application
4. THE Platform SHALL support database read replicas for reporting queries to reduce load on the primary database

### Requirement 83: Configurable Notification Preferences

**User Story:** As an Org_Admin, I want all notifications and reminders to be individually configurable, so that I control exactly what my organisation sends and when.

#### Acceptance Criteria

1. THE Notification_Module SHALL present all notification types as individually toggleable settings (disabled by default)
2. WHEN an Org_Admin enables a notification type, THE Notification_Module SHALL send that notification automatically until the Org_Admin disables it
3. THE Notification_Module SHALL group notification settings by category (Invoicing, Payments, Vehicle Reminders, System Alerts) in the settings interface for easy management
4. THE Notification_Module SHALL allow Org_Admin to configure the sending channel (email, SMS, or both) independently for each notification type that supports multiple channels
