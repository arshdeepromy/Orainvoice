# Requirements Document

Cloud Backup & Restore (Platform DR/BCP)

## Introduction

This feature provides a platform-wide Disaster Recovery / Business Continuity (DR/BCP) system for OraInvoice. It is configured and operated **exclusively from the Global Admin tier** — the platform-level administration scope that sits above all customer organisations. No organisation-level role has any access to this feature.

The system produces **full PostgreSQL dumps** (pg_dump custom format) that capture the entire database — all schemas, tables, sequences, feature flags, integration settings, and customer data across every organisation, with nothing excluded. In addition to the database dump, each backup captures the **uploaded files held on the filesystem** (customer/invoice/quote/job-card attachments, expense receipts, staff/payroll documents and photos, page-editor media, and compliance documents) so that file references stored in the database do not become dangling pointers after a restore. Database-stored (BYTEA) assets such as organisation branding images travel inside the dump itself and are not captured separately as files. Files are captured incrementally using content addressing so that unchanged files are not re-uploaded on every run. Backups are encrypted client-side before upload to a connected third-party cloud destination (Google Drive or OneDrive) and can be restored onto a fresh deployment, including a deployment that has only just connected to a cloud account that already holds backups.

The system also supports **per-organisation restore** from a full backup: a Global Admin can browse a backup's contents, select a single organisation (and optionally specific entities within that organisation), and restore that organisation's data without affecting other organisations.

### Distinction from `byo-drive-backup`

This feature is **distinct from and must not replace** the existing `byo-drive-backup` spec. The key differences:

| Aspect | `byo-drive-backup` (existing) | `cloud-backup-restore` (this feature) |
|---|---|---|
| Scope | Per-organisation, org-scoped | Platform-wide, all organisations |
| Operated by | Org admin / owner | Global Admin only |
| Backup content | Per-org DB slice (COPY by `org_id`) + per-org files | Full `pg_dump` custom-format dump of the entire database (including DB-stored BYTEA assets such as branding) + content-addressed incremental file blobs (uploads, payroll/HR media, page-editor media, compliance docs) |
| Destination | Each org's own Google Drive | Platform-connected Google Drive / OneDrive |
| Restore | Selective per-customer within one org | Full restore, per-org restore, or selective entities |
| Purpose | Org-owned data ownership / recovery | Platform DR/BCP |

This feature reuses the **encryption, OAuth, notification, audit, and scheduler patterns** established by `byo-drive-backup` and the wider codebase to stay consistent, but operates as an independent platform-level system.

## Glossary

- **Global_Admin**: A user with `role = 'global_admin'`, the platform-level administrator above all customer organisations. The only role permitted to configure or operate this feature.
- **Backup_System**: The overall cloud backup and restore subsystem described by this document.
- **Full_Backup**: A single completed backup artifact produced by `pg_dump` in custom format, capturing the entire PostgreSQL database (all schemas, tables, sequences, settings, feature flags, and customer data, including DB-stored BYTEA assets), together with the accompanying content-addressed File_Blobs and their File_Index. The database dump artifact and every File_Blob are encrypted client-side under the Backup_Data_Key hierarchy (a Backup_Data_Key wrapped by the escrowed Backup_Master_Key), NOT under the deployment's `ENCRYPTION_MASTER_KEY`, so that the escrowed key material can decrypt the backup on a fresh deployment where `ENCRYPTION_MASTER_KEY` is lost. The encrypted artifacts are stored in a connected cloud destination.
- **Uploaded_Files**: The set of files stored on the application filesystem (rather than inside the database) that are in scope for the file backup set, defined as every filesystem-backed file under the two storage roots `/app/uploads/` (all category subfolders) and `/app/compliance_files/`. This set is captured wholesale rather than via a hardcoded category allowlist, so that new upload categories are included automatically as they are added. The known filesystem-backed categories comprise: quote attachments (`quote-attachments`), invoice attachments (`invoice-attachments`), job-card attachments (`job-card-attachments`), expense receipts (`receipts`), staff/employment-agreement and generic documents (`attachments`), clock-in photos (`clock_photos`, kiosk biometric — see Requirement 20), staff profile/passport photos (`staff_photos`), payslip PDFs (`payslips`, payroll PII), CMS/page-editor media (`page-editor`), and compliance documents under `/app/compliance_files/`. Most categories use the path shape `{category}/{org_id}/{file}` and are naturally org-partitioned. **Database-stored (BYTEA) assets are explicitly OUT OF SCOPE for the Uploaded_Files set because they are already captured inside the `pg_dump` database dump.** In particular, organisation branding images (logo, dark_logo, favicon) are authoritatively stored as BYTEA columns on the `platform_branding` table (migration 0165) — a startup hook (`_migrate_branding_files_to_db`) migrates any legacy disk files into the database — so branding bytes travel inside the dump together with their org-owning row, and the legacy `/app/uploads/branding/` directory is a pre-migration copy that is NOT separately backed up. Other BYTEA columns (encrypted secrets/tokens) are likewise inside the dump. Each filesystem-backed file is referenced from the database by a `file_key` or path column.
- **File_Blob**: An individual Uploaded_File stored in the Cloud_Provider as a separately-encrypted, content-addressed object. Each File_Blob is named by a keyed hash (HMAC-SHA-256 under a platform secret) of the file's plaintext content so that identical content deduplicates within the platform's own File_Index without exposing plaintext equality to the Cloud_Provider, and each File_Blob is individually encrypted client-side (`envelope_encrypt`) under the Backup_Data_Key hierarchy (NOT under `ENCRYPTION_MASTER_KEY`) before upload, so that the escrowed Backup_Master_Key can decrypt it on a fresh deployment.
- **Content_Hash**: The SHA-256 hash of a file's plaintext content, used by the Backup_System for change detection (deciding whether a File_Blob already exists) and deduplication within the platform.
- **File_Index**: A per-Full_Backup metadata document that maps each captured file reference (its `file_key` or path, plus the owning `org_id`) to that file's Content_Hash and byte size. The File_Index is the authoritative list of which File_Blobs belong to a given Full_Backup, replacing the former single-bundle file archive. Because the per-file `file_key`/path and `org_id` listing leaks organisation structure and filenames to the Cloud_Provider as plaintext metadata, the File_Index path/org listing is stored INSIDE the encrypted envelope (encrypted at rest under the Backup_Data_Key), separated from the small set of cleartext catalog fields the provider/listing must read (see Requirement 7.2).
- **File_Archive**: Deprecated single-bundle term retained only for historical reference. The file portion of a Full_Backup is no longer a single bundled archive; it is realized as the set of content-addressed File_Blobs (each encrypted under the Backup_Data_Key hierarchy) referenced by that backup's File_Index (see Requirements 21, 22, and 24).
- **Backup_Master_Key**: A backup-specific master key that is escrowed/exported out-of-band (independently of the deployment being backed up) and re-supplied to a fresh deployment at restore time, typically by an operator passphrase that unwraps a stored Backup_Data_Key. The Backup_Master_Key is the root of trust for decrypting backups after a total loss of the original deployment.
- **Backup_Data_Key**: The per-backup (or per-key-version) data-encryption key that actually encrypts backup artifacts (the `pg_dump` database dump), the File_Blobs, and the encrypted portion of the File_Index. The Backup_Data_Key is stored only in wrapped form, wrapped by the Backup_Master_Key (or by an external KMS key where available), so that possession of the Backup_Master_Key is sufficient to decrypt backups on a fresh deployment. The Backup_Data_Key hierarchy is the ONLY key system used to encrypt backup artifacts; it is deliberately separate from the deployment's `ENCRYPTION_MASTER_KEY` (see ENCRYPTION_MASTER_KEY).
- **ENCRYPTION_MASTER_KEY**: The deployment-local operational master key used by the existing `app/core/encryption` helpers to encrypt operational secrets that live on the running deployment — for example stored OAuth access/refresh tokens (Requirement 2.4) and other integration credentials. It is regenerated/re-provisioned on a fresh deployment and is therefore NOT assumed to survive a disaster; for that reason it MUST NOT be used to encrypt backup artifacts or File_Blobs (those use the escrowed Backup_Data_Key hierarchy). The two key systems are intentionally distinct and MUST NOT be conflated: `ENCRYPTION_MASTER_KEY` protects operational secrets in place, the Backup_Data_Key hierarchy protects portable backup artifacts for DR.
- **Backup_Scope**: The Global-Admin-selected content of a backup. One of: `settings_only` (global admin settings plus all integration settings), `organisations_only` (all customer organisations and their data), or `both`.
- **Cloud_Provider**: A configured backup storage destination (also referred to as a Storage_Destination). Supported destination types in this version are `google_drive`, `onedrive`, `nas`, and `s3` (S3-compatible). Destination types differ in how they authenticate: `google_drive` and `onedrive` are OAuth cloud drives (authenticated per Requirement 2); `s3` is S3-compatible object storage — AWS S3 or a compatible endpoint such as MinIO, Backblaze B2, or Wasabi — authenticated by access keys (Requirement 28); and `nas` is network-attached storage reached as an SMB/CIFS or NFS network share or a mounted volume path, authenticated by share credentials or mount configuration (Requirement 29). NAS and S3 are therefore Cloud_Provider types that authenticate differently (NOT OAuth). Every destination type is accessed through the same provider-agnostic Storage_Interface (Requirement 3), so backup and restore logic is identical regardless of destination type.
- **Storage_Interface**: The provider-agnostic abstraction through which the Backup_System uploads, lists, downloads, and deletes backup artifacts, allowing additional destination types (for example NAS and S3, in addition to the OAuth cloud drives) to be added without changing backup or restore logic.
- **OAuth_Connection**: A stored, encrypted authorization to an OAuth Cloud_Provider account (`google_drive` or `onedrive`), including access and refresh tokens, configured independently per provider from Global Admin settings. NAS_Destination and S3_Destination do not use an OAuth_Connection; they authenticate via share credentials or access keys respectively (Requirements 28 and 29).
- **NAS_Destination**: A Cloud_Provider of type `nas` — a network-attached storage target reached as an SMB/CIFS or NFS network share, or as a mounted volume path. A NAS_Destination is configured by a network share path / mount target plus any credentials required to access it (for example an SMB username and password), all stored envelope-encrypted under `ENCRYPTION_MASTER_KEY` (operational secrets, the same protection used for OAuth tokens in Requirement 2.4 — NOT the Backup_Data_Key hierarchy). The Backup_System writes the encrypted backup artifact set (database dump, File_Blobs, File_Index, and manifests) into a configured directory on the share through the Storage_Interface using an atomic temp-file-then-rename write. A NAS sited in New Zealand can keep encrypted backups onshore (see Requirement 20). A standard NAS share provides no true WORM/immutability and is therefore not a substitute for the Immutable_Copy (see Requirement 27).
- **S3_Destination**: A Cloud_Provider of type `s3` — AWS S3 or any S3-compatible object storage (for example MinIO, Backblaze B2, or Wasabi). An S3_Destination is configured by an access key ID and secret access key (and an optional session token), a bucket name, a region, an optional endpoint URL (for S3-compatible providers), and an optional addressing style (path-style vs virtual-hosted), with the credentials stored envelope-encrypted under `ENCRYPTION_MASTER_KEY` (operational secrets, the same protection used for OAuth tokens in Requirement 2.4 — NOT the Backup_Data_Key hierarchy). The Backup_System reads and writes the encrypted backup artifact set through the Storage_Interface using S3 multipart upload for large artifacts. An S3 bucket in a New Zealand region, or a self-hosted onshore S3-compatible endpoint, can keep encrypted backups onshore (see Requirement 20); an S3 bucket with Object Lock in compliance mode is the natural Immutable_Copy destination (see Requirement 27).
- **Backup_Manifest**: A metadata document accompanying each Full_Backup, recording the timestamp, application version, database schema/migration version, backup size, checksum, included Backup_Scope, the list of organisation IDs contained, the Uploaded_Files summary (file count and total file bytes), and the per-backup File_Index (the list of file references with their Content_Hash and byte size).
- **Backup_Job**: A tracked unit of work that produces a Full_Backup, with a status lifecycle and progress information.
- **Restore_Job**: A tracked unit of work that restores from a Full_Backup, with a status lifecycle, progress information, and human-readable error reporting.
- **Pre_Restore_Snapshot**: A safety backup of the current target database state taken automatically before a full restore begins, used for automatic rollback if the restore fails validation.
- **Maintenance_Mode**: A platform state during which normal application traffic is blocked or held while a full restore is in progress.
- **Dry_Run**: A validation-only execution of a restore that performs all checks and reports the planned outcome without committing changes to the target database.
- **Retention_Policy**: The Global-Admin-configured rules governing how many backups are kept and for how long, after which expired backups are pruned from cloud storage.
- **Backup_Window**: An optional Global-Admin-configured time range within which scheduled backups are permitted to start.
- **Tenant_Isolation_Strategy**: The multi-tenancy model used to associate rows with an organisation. OraInvoice uses row-level isolation via an `org_id` column with PostgreSQL Row-Level Security (RLS). (See Open Questions for schema-per-tenant considerations.)
- **Schema_Compatibility_Check**: A pre-restore comparison of the Backup_Manifest's database migration version against the target deployment's migration version, used to refuse or warn before restoring an incompatible backup.
- **Webhook_Notification**: An HTTP POST to a Global-Admin-configured external URL carrying a defined payload describing a backup or restore event, used for external monitoring.
- **Connexus**: The SMS provider integration used by OraInvoice for outbound SMS notifications.
- **Audit_Log**: The existing `audit_logs` table used to record every backup and restore action with actor and timestamp.
- **Per_Org_Index**: A lightweight per-organisation index/manifest emitted at backup time alongside the database dump, containing for each `org_id` a summary of its entities (per-entity-type record counts and identifiers) and optionally a Per_Org_Logical_Export, so that per-organisation browsing (Requirement 15) can be served without staging the full custom-format `pg_dump`. The Per_Org_Index is part of the Backup_Manifest set and its org-identifying contents are encrypted under the Backup_Data_Key hierarchy. The Per_Org_Index also records, per `org_id`, whether a Per_Org_Logical_Export was emitted for that organisation in this Full_Backup (see Requirement 31).
- **Per_Org_Logical_Export**: An optional per-organisation logical export of a single organisation's Org_Scoped_Rows (the org's tenant rows emitted as a `COPY`/SQL representation keyed by `org_id`) emitted at backup time alongside the full `pg_dump` dump, so that a per-organisation restore can read only that organisation's data without staging the full dump. A Per_Org_Logical_Export is emitted opportunistically per Requirement 31 only WHERE the backup size/time budget allows; the full dump remains the unconditional system-of-record. Like every other backup artifact, a Per_Org_Logical_Export is encrypted client-side under the Backup_Data_Key hierarchy and its org-identifying contents reside inside the encrypted envelope (consistent with Requirement 7.8). It is NOT a substitute for the full dump and does not change the unconditional presence of the full dump in every backup.
- **Org_Scoped_Row**: A database row that is tenant data belonging to exactly one organisation, identified by carrying an `org_id` column. Org_Scoped_Rows are the rows a per-organisation restore inserts, updates, or deletes, and are the rows whose cross-org modification is prohibited by Requirement 14.3.
- **Shared_Global_Row**: A database row that is shared/global platform reference data rather than tenant data — a row that has no `org_id` column OR that belongs to the enumerated set of global reference tables (for example subscription plans, global vehicle/CarJam records, and feature flags). Shared_Global_Rows are not owned by any single organisation; during a per-organisation restore they are subject to the read-and-ensure-exists handling of Requirement 14.7 (inserted only if absent in the target, never modified or deleted) rather than the cross-org prohibition of Requirement 14.3.
- **Streaming_Replication**: The PostgreSQL streaming-replication relationship operated by CloudNativePG between a primary instance and one or more live read-replica Standby_Node instances. Restoring a full dump into a primary that is still streaming to a live standby can corrupt the standby or trigger an unintended failover, so a full restore must isolate the primary first (see Requirement 12).
- **Standby_Node**: A PostgreSQL instance that receives streaming replication from the primary under CloudNativePG. During a full restore the standby must be detached/fenced and later re-seeded from the restored primary.
- **Recovery_Point_Objective (RPO)**: The maximum acceptable amount of data loss measured as a duration of time (the maximum age of the most recent recovery point). Backup frequency must be configured so the interval between successful backups does not exceed the RPO.
- **Recovery_Time_Objective (RTO)**: The maximum acceptable duration to restore service after a disaster is declared. The restore design and rehearsal results must demonstrate recovery within the RTO.
- **Restore_Rehearsal**: An automated, scheduled test that restores a recent Full_Backup into an isolated scratch environment, runs validation (schema + row-count + file-consistency + an application smoke check), records a pass/fail result, and tears the scratch environment down. Distinct from Dry_Run, which only validates checksum and schema and never actually applies the dump.
- **Immutable_Copy**: An optional second backup destination configured with delete-protection / object-lock / write-once-read-many (WORM) immutability for a defined lock window, such that prune/retention/GC operations cannot delete a backup within its lock window even with a compromised Global_Admin credential.
- **Orphan_Blob**: A File_Blob present in the Cloud_Provider that is referenced by no committed Full_Backup File_Index — typically left behind by a backup that uploaded blobs but failed before writing its manifest. Orphan_Blobs are invisible to reference-counted pruning and are reclaimed by the mark-and-sweep garbage collection in Requirement 8.

## Requirements

### Requirement 1: Global-Admin-Only Access Control

**User Story:** As a Global Admin, I want the cloud backup and restore feature to be accessible only from the Global Admin tier, so that platform-wide DR operations cannot be triggered or viewed by any organisation user.

#### Acceptance Criteria

1. WHEN a request is received on any backup or restore API endpoint, THE Backup_System SHALL authorise the request only if the authenticated user's role is exactly `global_admin`, and SHALL deny it for every other role.
2. IF a request to any backup or restore API endpoint carries a valid authentication token whose user role is not `global_admin`, THEN THE Backup_System SHALL reject the request with HTTP 403, SHALL NOT execute the requested backup or restore action, and SHALL leave all backup data and configuration unchanged.
3. IF a request to any backup or restore API endpoint carries no authentication token or an invalid or expired token, THEN THE Backup_System SHALL reject the request with HTTP 401 and SHALL NOT execute the requested action.
4. WHEN the application interface is rendered for a user whose role is not `global_admin`, THE Backup_System SHALL NOT display any backup or restore configuration, control, or navigation entry, and SHALL expose these elements only when the user's role is `global_admin`.
5. WHEN a `global_admin` user performs a backup configuration change, a backup action, or a restore action, THE Backup_System SHALL write one record to the Audit_Log containing the acting user identifier, the action type, the target resource identifier, and the UTC timestamp of the action.
6. IF a request to any backup or restore API endpoint is rejected for failing the `global_admin` authorisation check, THEN THE Backup_System SHALL write one record to the Audit_Log containing the requesting user identifier (or an unauthenticated indicator when no valid token is present), the attempted action type, and the UTC timestamp of the attempt.

### Requirement 2: Cloud Provider Connection via OAuth

**User Story:** As a Global Admin, I want to connect Google Drive and OneDrive accounts using OAuth, so that the platform can store encrypted backups in cloud storage I control.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to configure each Cloud_Provider connection (`google_drive` and `onedrive`) independently from Global Admin settings, such that configuring or removing one provider has no effect on the stored credentials or connection state of the other.
2. WHEN a Global_Admin initiates a connection to a Cloud_Provider, THE Backup_System SHALL perform an OAuth 2.0 authorization-code flow requesting offline access so that a refresh token is obtained, and SHALL set the connection state to `connected` only after a refresh token is successfully received and stored.
3. IF the OAuth 2.0 authorization-code flow does not complete with both an access token and a refresh token within 300 seconds of initiation, OR the Cloud_Provider returns an authorization error, THEN THE Backup_System SHALL leave the connection in the `disconnected` state, persist no partial OAuth credentials, and present an error indication to the Global_Admin describing that the connection was not established.
4. THE Backup_System SHALL store OAuth access tokens and refresh tokens envelope-encrypted using the existing `app/core/encryption` helpers with the `ENCRYPTION_MASTER_KEY`, and SHALL never persist these tokens in unencrypted form. (These are operational secrets that live on the running deployment and are re-provisioned on a fresh deploy; they are deliberately protected by `ENCRYPTION_MASTER_KEY`, which is a distinct key system from the escrowed Backup_Data_Key hierarchy used for backup artifacts per Requirement 16 — the two MUST NOT be conflated.)
5. WHEN an access token is expired or expires within 60 seconds of a pending cloud operation, THE Backup_System SHALL obtain a new access token using the stored refresh token before performing that cloud operation.
6. IF a Cloud_Provider rejects the stored refresh token as revoked or invalid, THEN THE Backup_System SHALL set the connection state to `disconnected`, dispatch the configured failure notification exactly once per disconnection event, and stop scheduling backups to that provider until the Global_Admin reconnects it.
7. WHEN a Global_Admin disconnects a Cloud_Provider, THE Backup_System SHALL revoke the stored tokens with the provider where provider revocation is supported, SHALL delete the stored OAuth credentials for that provider, and SHALL set the connection state to `disconnected`.
8. THE Backup_System SHALL exclude OAuth access tokens and refresh tokens from all log output, including error and debug logs.

### Requirement 3: Provider-Agnostic Storage Interface

**User Story:** As a platform engineer, I want backup storage operations to go through a provider-agnostic interface, so that a new provider such as S3 can be added later without modifying backup or restore logic.

#### Acceptance Criteria

1. THE Backup_System SHALL perform every upload, list, download, and delete operation exclusively through the Storage_Interface, and SHALL NOT invoke any Cloud_Provider SDK or API directly from backup or restore logic.
2. THE Storage_Interface SHALL define a uniform contract consisting of exactly five operations — upload, list, download, delete, and connection-status — where each operation accepts and returns provider-independent values (no parameter or return type references a specific Cloud_Provider).
3. THE Storage_Interface connection-status operation SHALL return one of the defined states `connected`, `disconnected`, or `error`, independent of which Cloud_Provider implementation is active.
4. WHERE a Cloud_Provider implementation conforms to all five Storage_Interface operations, THE Backup_System SHALL support that provider with no changes to backup creation logic or restore logic.
5. WHEN the Backup_System initializes, THE Backup_System SHALL select the active Cloud_Provider implementation identified by the current Global-Admin configuration value.
6. IF the Global-Admin configuration identifies no provider or identifies a provider that has no registered Storage_Interface implementation, THEN THE Backup_System SHALL reject the operation, return an error indicating the provider is unavailable, and SHALL NOT execute any upload, list, download, or delete.
7. IF a Storage_Interface operation fails at the active Cloud_Provider, THEN THE Backup_System SHALL return a uniform error result identifying the failed operation and SHALL preserve the prior backup state without partial modification.
8. WHERE the active Cloud_Provider is a NAS_Destination (Requirement 29) or an S3_Destination (Requirement 28), THE Backup_System SHALL access it exclusively through the same five Storage_Interface operations (upload, list, download, delete, and connection-status) used for the OAuth Cloud_Providers, with no change to backup creation logic or restore logic.
9. THE Backup_System SHALL operate the content-addressed File_Blob model, the Backup_Manifest, the File_Index, reference-counted pruning, and garbage collection (Requirements 7, 8, 21, and 24) identically regardless of which Cloud_Provider destination type (`google_drive`, `onedrive`, `nas`, or `s3`) is active.

### Requirement 4: Resumable Chunked Uploads

**User Story:** As a Global Admin, I want multi-gigabyte backups to upload reliably over unstable networks, so that a transient interruption does not force the entire upload to restart.

#### Acceptance Criteria

1. WHEN the Backup_System uploads a backup artifact, THE Backup_System SHALL use the Cloud_Provider's resumable or chunked upload mechanism (Google Drive resumable upload, OneDrive upload session) with a chunk size between 5 MiB and 100 MiB, defaulting to 16 MiB and constrained to a multiple of 256 KiB.
2. WHEN a chunk is acknowledged by the Cloud_Provider, THE Backup_System SHALL persist the last acknowledged byte offset before sending the next chunk.
3. IF a chunk upload fails due to a transient network error (a request timeout exceeding 30 seconds, a connection reset, or a retriable or rate-limit response from the Cloud_Provider), THEN THE Backup_System SHALL retry the failed chunk using exponential backoff with an initial delay of 1 second, a multiplier of 2, a maximum delay of 60 seconds, and up to 1 second of jitter, for up to 5 attempts per chunk.
4. IF a chunk upload fails due to a non-transient error (authentication rejection, storage quota exceeded, or an invalid-request response), THEN THE Backup_System SHALL stop the upload immediately, mark the Backup_Job as `failed`, remove any partially uploaded artifact from the Cloud_Provider, and dispatch the configured failure notification.
5. WHEN an interrupted upload is resumed, THE Backup_System SHALL query the Cloud_Provider for the last acknowledged byte offset and resume the upload from that offset rather than restarting from the beginning.
6. IF the upload session is expired or invalid, THEN THE Backup_System SHALL create a new upload session, restart the upload from byte offset 0, record the restart, and retry session creation up to 3 times before marking the Backup_Job as `failed`.
7. IF a chunk does not upload successfully after the maximum retry attempts, THEN THE Backup_System SHALL mark the Backup_Job as `failed`, remove any partially uploaded artifact from the Cloud_Provider, and dispatch the configured failure notification.
8. WHERE the active Cloud_Provider is an S3_Destination (Requirement 28), THE Backup_System SHALL upload large artifacts using S3 multipart upload, applying the same resume, retry, and exponential-backoff semantics defined in criteria 2–7 (persisting each acknowledged part, resuming from the last acknowledged part rather than restarting from the beginning, and retrying a transient part failure up to 5 attempts), using a part size within the 5 MiB–100 MiB bound of criterion 1 subject to the S3 minimum part size.
9. WHERE the active Cloud_Provider is a NAS_Destination (Requirement 29), THE Backup_System SHALL write each artifact to a temporary file in the configured directory on the share and SHALL atomically rename it to its final path only after the complete artifact has been written, and IF a NAS write fails due to a transient error THEN THE Backup_System SHALL retry the write using the exponential-backoff schedule of criterion 3; THE Backup_System MAY write a NAS artifact without chunking, because local/LAN writes need not be chunked, but every NAS artifact write SHALL be atomic (temp-file-then-rename) and retryable.
10. IF an S3 multipart upload session or part is invalid or expired, OR a NAS atomic rename fails, THEN THE Backup_System SHALL apply the non-transient-failure handling of criteria 4 and 7 — stop the upload, mark the Backup_Job as `failed`, remove any partially written artifact (an incomplete multipart upload or a leftover temporary file) from the destination, and dispatch the configured failure notification.

### Requirement 5: Full PostgreSQL Backup Creation

**User Story:** As a Global Admin, I want backups to be full PostgreSQL dumps capturing everything, so that a restore can rebuild the entire platform on a fresh deployment.

#### Acceptance Criteria

1. WHEN a Backup_Job runs, THE Backup_System SHALL produce a Full_Backup using `pg_dump` in custom format that includes every non-template database object (all schemas, tables, sequences, views, indexes, constraints, and row data), covering feature flags, integration settings, and customer data, such that no object present in the source database is omitted from the dump.
2. WHILE the Backup_Scope is `both`, THE Backup_System SHALL include all application schemas and all application table rows in the Full_Backup, excluding zero application data objects.
3. WHEN a Full_Backup is produced, THE Backup_System SHALL encrypt the complete Full_Backup artifact client-side under the Backup_Data_Key hierarchy (the per-backup Backup_Data_Key wrapped by the escrowed Backup_Master_Key, as defined in Requirement 16), NOT under the deployment's `ENCRYPTION_MASTER_KEY`, before any byte of the artifact is transmitted to the Cloud_Provider, such that the unencrypted artifact is never uploaded.
4. WHEN a Full_Backup is produced, THE Backup_System SHALL generate exactly one Backup_Manifest recording the artifact identifier, byte size, checksum, creation timestamp (UTC, ISO 8601), source database name, Backup_Scope, the included Uploaded_Files count, and the total Uploaded_Files bytes, and SHALL store the Backup_Manifest in the same Cloud_Provider location as the backup artifact.
5. WHEN a Full_Backup completes with `pg_dump` exiting with status code 0 and the encrypted artifact and Backup_Manifest both confirmed stored in the Cloud_Provider, THE Backup_System SHALL set the Backup_Job status to `succeeded` and record the artifact location (Cloud_Provider path), the artifact byte size, and the artifact checksum.
6. IF `pg_dump` exits with a non-zero status code, THEN THE Backup_System SHALL set the Backup_Job status to `failed`, retain a human-readable error description indicating the failure cause, leave any prior successful Full_Backup artifacts in the Cloud_Provider unchanged, and dispatch the configured failure notification within 60 seconds of detecting the non-zero exit.
7. IF the encrypted Full_Backup artifact or the Backup_Manifest fails to upload to the Cloud_Provider, THEN THE Backup_System SHALL set the Backup_Job status to `failed`, retain a human-readable error description indicating the upload failure, and dispatch the configured failure notification within 60 seconds of detecting the upload failure.
8. WHERE the Backup_Scope is `organisations_only` or `both`, THE Backup_System SHALL include the Uploaded_Files in the produced backup as a set of content-addressed File_Blobs referenced by the Full_Backup's File_Index, each File_Blob encrypted client-side under the Backup_Data_Key hierarchy (the same key system as the database artifact, NOT `ENCRYPTION_MASTER_KEY`) before any byte is transmitted to the Cloud_Provider, as further specified in Requirement 21.
9. THE Backup_System SHALL produce the full `pg_dump` custom-format dump in every Full_Backup unconditionally, and WHERE the Backup_Scope is `organisations_only` or `both` and the backup size/time budget allows, THE Backup_System SHALL additionally emit a Per_Org_Logical_Export for one or more contained organisations alongside the full dump (per Requirement 31), encrypted client-side under the Backup_Data_Key hierarchy, such that the presence or absence of a Per_Org_Logical_Export never affects the unconditional presence of the full dump.

### Requirement 6: Selectable Backup Scope

**User Story:** As a Global Admin, I want to choose what a backup contains, so that I can run lightweight settings-only backups or full platform backups as needed.

#### Acceptance Criteria

1. WHEN a Global_Admin initiates a backup, THE Backup_System SHALL accept a Backup_Scope value of exactly one of the following: `settings_only`, `organisations_only`, or `both`.
2. IF the Backup_Scope value is absent, empty, or not one of `settings_only`, `organisations_only`, or `both`, THEN THE Backup_System SHALL reject the backup request, SHALL NOT create any backup artefact or Backup_Manifest, and SHALL return an error response indicating the Backup_Scope value is invalid and listing the accepted values.
3. WHERE the Backup_Scope is `settings_only`, THE Backup_System SHALL include all global admin settings and all integration settings in the backup, and SHALL exclude all customer organisations and their data.
4. WHERE the Backup_Scope is `organisations_only`, THE Backup_System SHALL include all customer organisations and their data in the backup, and SHALL exclude all global admin settings and integration settings.
5. WHERE the Backup_Scope is `both`, THE Backup_System SHALL include all global admin settings, all integration settings, and all customer organisations and their data in the backup.
6. WHEN determining which settings and integration settings are included for a backup with Backup_Scope of `settings_only` or `both`, THE Backup_System SHALL derive the included items from the application's settings and integration configuration sources at runtime so that every setting present in those sources at backup time is included.
7. THE Backup_System SHALL NOT use a hardcoded allowlist of setting names to determine backup inclusion.
8. WHEN a backup completes successfully, THE Backup_System SHALL record the selected Backup_Scope value in the Backup_Manifest.

### Requirement 7: Backup Manifest and Integrity

**User Story:** As a Global Admin, I want each backup to carry a verifiable manifest, so that I can confirm a backup's contents and integrity before restoring it.

#### Acceptance Criteria

1. THE Backup_Manifest SHALL include all of the following fields, each non-empty: the backup creation timestamp in ISO 8601 UTC format, the application version, the database schema/migration version, the backup size in bytes (integer ≥ 0), a checksum of the encrypted backup artifact, the included Backup_Scope, the list of organisation IDs contained in the backup (a list with 0 or more entries), the Uploaded_Files count (integer ≥ 0), and the total Uploaded_Files bytes (integer ≥ 0).
2. THE Backup_Manifest SHALL include a per-backup File_Index recording, for every captured Uploaded_File, its `file_key` or path, its owning `org_id` (or an explicit global/non-org indicator), its Content_Hash, and its byte size (integer ≥ 0). Because the per-file `file_key`/path and `org_id` listing reveals organisation structure and filenames, THE Backup_System SHALL store the File_Index path/org listing fields (the `file_key`/path and owning `org_id` of each entry) INSIDE the encrypted envelope, encrypted at rest under the Backup_Data_Key hierarchy, so that those fields are not exposed to the Cloud_Provider as plaintext. The database dump remains a full dump in every backup and is not subject to the File_Index.
3. WHEN a Full_Backup is created, THE Backup_System SHALL compute a checksum over the complete encrypted backup artifact and record that checksum value in the Backup_Manifest before the backup is marked as completed.
4. WHEN a backup artifact is selected for restore, THE Backup_System SHALL recompute the checksum over the downloaded encrypted artifact and compare it byte-for-byte against the checksum recorded in the Backup_Manifest before initiating any restore operation.
5. IF the Backup_Manifest is missing, unreadable, or does not contain a recorded checksum when a restore is requested, THEN THE Backup_System SHALL abort the restore before any data is modified, mark the Restore_Job as `failed` with an integrity error indicating the manifest is invalid or missing, and dispatch the configured failure notification.
6. IF the recomputed checksum does not match the checksum recorded in the Backup_Manifest, THEN THE Backup_System SHALL abort the restore before any data is modified, leave the target data unchanged, mark the Restore_Job as `failed` with an integrity error indicating a checksum mismatch, and dispatch the configured failure notification.
7. WHEN the recomputed checksum matches the checksum recorded in the Backup_Manifest, THE Backup_System SHALL allow the restore operation to proceed.
8. THE Backup_System SHALL keep readable to the Cloud_Provider and to the listing/discovery flow ONLY the small set of catalog fields needed to list and select a backup — namely the backup identifier, the backup creation timestamp (ISO 8601 UTC), the encrypted-artifact byte size, the artifact checksum, and the included Backup_Scope — and SHALL store every other Backup_Manifest field that reveals customer or organisation structure (the list of organisation IDs, the File_Index path/org listing, and the Per_Org_Index org-identifying contents) inside the encrypted envelope under the Backup_Data_Key hierarchy, such that no organisation identifier, file path, or filename is written in cleartext at the Cloud_Provider.
9. WHEN a Full_Backup is produced with Backup_Scope `organisations_only` or `both`, THE Backup_System SHALL also produce a Per_Org_Index emitted at backup time that records, for each contained `org_id`, a per-entity-type summary (record counts and identifiers) sufficient to serve per-organisation browsing (Requirement 15) without staging the full custom-format `pg_dump`, SHALL record in the Per_Org_Index, for each contained `org_id`, whether a Per_Org_Logical_Export was emitted for that organisation in this Full_Backup (and, where emitted, the location of that organisation's Per_Org_Logical_Export artifact), and SHALL store the Per_Org_Index in the same Cloud_Provider location as the Backup_Manifest with its org-identifying contents encrypted under the Backup_Data_Key hierarchy.

### Requirement 8: Schedule, Retention, and Pruning

**User Story:** As a Global Admin, I want to schedule automatic backups and control how long backups are kept, so that recovery points are maintained without unbounded cloud storage growth.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to configure backup frequency using a cron-style schedule expressed in the New Zealand timezone.
2. THE Backup_System SHALL allow a Global_Admin to configure an optional Backup_Window during which scheduled backups are permitted to start.
3. WHERE a Backup_Window is configured, THE Backup_System SHALL start a scheduled Backup_Job only within that window.
4. THE Backup_System SHALL allow a Global_Admin to configure a Retention_Policy specifying both the number of backups to keep and a retention period.
5. WHEN a backup's age exceeds the configured retention period, THE Backup_System SHALL delete that backup from the Cloud_Provider by removing its database dump and its File_Index, and SHALL apply the reference-counted File_Blob pruning rule in criterion 9 to that backup's referenced File_Blobs.
6. WHEN the number of stored backups exceeds the configured number to keep, THE Backup_System SHALL delete the oldest backups from the Cloud_Provider (removing each deleted backup's database dump and File_Index, and applying the reference-counted File_Blob pruning rule in criterion 9) until the count is within the configured limit.
7. IF deletion of an expired backup from the Cloud_Provider fails, THEN THE Backup_System SHALL retain the backup record, mark it with a pruning-failed status, log the failure, and retry pruning on the next scheduled cycle.
8. THE Backup_System SHALL allow a Global_Admin to trigger a manual on-demand backup in addition to scheduled backups.
9. WHEN a backup is deleted (by retention age or by count limit), THE Backup_System SHALL delete a content-addressed File_Blob only when no retained backup's File_Index references that File_Blob's Content_Hash, and SHALL NOT delete or orphan any File_Blob whose Content_Hash is still referenced by at least one retained backup's File_Index.
10. THE Backup_System SHALL run a mark-and-sweep File_Blob garbage collection (independent of reference-counted pruning) that enumerates the File_Blobs stored at the Cloud_Provider, identifies each Orphan_Blob that is referenced by no committed Full_Backup File_Index (for example a blob left behind by a failed or partial backup that uploaded blobs but wrote no manifest, per Requirements 4.4, 5.6, 5.7, and 21.10), and SHALL delete an Orphan_Blob only after that blob has been continuously unreferenced for a configurable safety grace period (default 24 hours), so that orphaned blobs cannot accumulate without bound.
11. WHILE a Backup_Job is in progress, THE Backup_System SHALL enforce mutual exclusion between backup creation and any prune or garbage-collection run for the same Cloud_Provider (for example by holding a prune/GC lock for the duration of the Backup_Job) OR SHALL apply a grace period such that no File_Blob is deleted while any Backup_Job that may reference it is in progress, so that a File_Blob skipped by the dedup rule (criterion 21.3, "upload only if absent") cannot be deleted by a concurrent prune or GC before the in-progress backup commits its manifest.
12. WHEN a Backup_Job reuses an existing File_Blob under the dedup rule (criterion 21.3) rather than re-uploading it, THE Backup_System SHALL verify and re-assert that File_Blob's presence at the Cloud_Provider at backup-commit time, and IF the reused File_Blob is found absent at commit time, THEN THE Backup_System SHALL re-upload the missing File_Blob before committing the manifest or, if it cannot, SHALL mark the Backup_Job as `failed` so that no committed manifest ever references a deleted blob.
13. WHEN a Global_Admin configures the backup frequency or the Retention_Policy, THE Backup_System SHALL validate the configuration against the configured Recovery_Point_Objective and Recovery_Time_Objective (Requirement 25) such that the interval between successive scheduled backups does not exceed the Recovery_Point_Objective, and IF the configured frequency or retention does not satisfy the configured objectives, THEN THE Backup_System SHALL present a warning to the Global_Admin identifying the unmet objective before the configuration is saved.

> **Design Decision — Orphan-blob reclamation and dedup-vs-prune safety.**
> Reference-counted pruning (criterion 9) only observes blobs referenced by a committed File_Index, so it can never reclaim blobs left by a backup that failed mid-upload before writing its manifest; those Orphan_Blobs would accumulate forever. A separate mark-and-sweep GC (criterion 10) closes that gap. Evaluated approaches to the dedup-vs-prune race: (i) a global prune/GC lock excluded for the duration of any in-progress Backup_Job; (ii) a grace period during which no blob is deleted while a backup that may reference it is running; (iii) commit-time re-assertion that every reused blob still exists. **Recommendation:** combine (i)/(ii) — exclude prune/GC while a backup runs, and apply the grace period of criterion 10 to all deletions — with (iii) commit-time re-assertion as a belt-and-braces guard (criterion 12). **Rationale:** "upload only if absent" (21.3) makes a blob skip its upload precisely when it already exists, which is exactly when a naive concurrent prune could delete it; mutual exclusion plus a grace period plus commit-time re-assertion guarantee a committed manifest never points at a deleted blob without forcing every backup to re-upload already-present content.

### Requirement 9: Restore from a Connected Cloud Account

**User Story:** As a Global Admin, I want to restore from a backup stored in a connected Google Drive or OneDrive account, so that I can recover the platform after data loss.

#### Acceptance Criteria

1. WHEN a Global_Admin opens the restore view for a connected Cloud_Provider account, THE Backup_System SHALL list the available backups within 30 seconds, displaying for each backup its filename, creation timestamp, and size, ordered by creation timestamp descending.
2. IF a connected Cloud_Provider account contains no available backups, THEN THE Backup_System SHALL present an empty-state indication that no backups are available for restore.
3. WHEN a Global_Admin connects a Cloud_Provider account that already contains backups, THE Backup_System SHALL discover and list those existing backups within 30 seconds of the connection completing.
4. IF backup discovery does not complete within 30 seconds, THEN THE Backup_System SHALL abort the discovery, present an error indication that the backups could not be listed, and allow the Global_Admin to retry.
5. IF backup discovery fails because the Cloud_Provider account is unreachable, the authorization is rejected, or access is denied, THEN THE Backup_System SHALL present a human-readable error indication and retain any previously listed backups.
6. THE Backup_System SHALL support restoring from a listed backup onto a fresh deployment that has only just connected the Cloud_Provider account, with no pre-existing organisation data required.
7. WHEN a restore is initiated, THE Backup_System SHALL create a Restore_Job whose status follows the lifecycle PENDING, then IN_PROGRESS, then exactly one terminal state of COMPLETED, FAILED, or CANCELLED (where CANCELLED is the terminal state reached when a pre-apply cancel succeeds per Requirement 12.16; this lifecycle maps to the `queued → running → completed | failed | cancelled` model of Requirement 13.1).
8. WHILE a Restore_Job is IN_PROGRESS, THE Backup_System SHALL report the restore progress as a percentage value between 0 and 100 inclusive.
9. WHEN a Restore_Job completes successfully, THE Backup_System SHALL set the Restore_Job to COMPLETED and report progress as 100 percent.
10. IF a restore operation fails, THEN THE Backup_System SHALL set the Restore_Job to FAILED, report a human-readable error description, and SHALL NOT expose raw stack traces in the user-facing error.

### Requirement 10: Schema-Version Compatibility Check

**User Story:** As a Global Admin, I want the system to check backup compatibility before restoring, so that I do not restore a backup whose schema is incompatible with the target deployment.

#### Acceptance Criteria

1. WHEN a restore is initiated, THE Backup_System SHALL read the database migration version (Alembic revision identifier) recorded in the Backup_Manifest and the target deployment's current migration version (Alembic revision identifier), and perform the Schema_Compatibility_Check before applying any data to the target database.
2. IF the Backup_Manifest does not contain a database migration version, or the recorded migration version is empty or does not correspond to a known Alembic revision in the target deployment, THEN THE Backup_System SHALL refuse the restore, apply no data to the target database, and return an error response indicating that the backup migration version is missing or unrecognized.
3. IF the backup migration version is newer than the target deployment's current migration version, THEN THE Backup_System SHALL refuse the restore, apply no data to the target database, and return an error response indicating schema incompatibility, naming both the backup migration version and the target migration version.
4. WHEN the Schema_Compatibility_Check determines the backup migration version is equal to the target deployment's current migration version, THE Backup_System SHALL proceed with the restore without requiring additional confirmation.
5. IF the backup migration version is older than the target deployment's current migration version, THEN THE Backup_System SHALL report this from the Dry_Run / pre-restore check as an older-schema warning naming both the backup migration version and the target migration version and indicating that explicit confirmation is required to proceed, and SHALL apply no data to the target database from the check itself.
6. WHEN a full restore is submitted for an older-schema backup AND the submission carries an explicit older-schema confirmation (the `confirm_older_schema` flag set true on the restore request), THE Backup_System SHALL proceed with the restore.
7. IF a full restore is submitted for an older-schema backup without an explicit older-schema confirmation, THEN THE Backup_System SHALL refuse the restore before applying any data, apply no data to the target database, return an error response that names both migration versions and indicates explicit confirmation is required, and record the restore decision as refused. (There is no in-flight pause or awaiting-confirmation job state: confirmation is obtained ahead of submission from the Dry_Run / pre-restore check result, then passed on the `POST /restore/full` request, so a status poll never needs to observe a paused job. A Global_Admin who declines the older-schema gate simply does not submit the confirmed request; where a restore was already started and is then cancelled by the operator before the destructive apply begins, it is recorded as cancelled per Requirement 12.)
8. WHEN the Schema_Compatibility_Check completes, THE Backup_System SHALL record in the Restore_Job the backup migration version, the target migration version, the comparison outcome (newer, equal, or older), and the final restore decision (proceeded, refused, or cancelled).

### Requirement 11: Dry-Run Validation

**User Story:** As a Global Admin, I want to validate a restore before committing it, so that I can confirm the restore will succeed without altering the target database.

#### Acceptance Criteria

1. WHERE the Global_Admin selects Dry_Run mode, THE Backup_System SHALL execute the restore validation without applying any write, schema change, or data modification to the target database.
2. WHEN a restore runs in Dry_Run mode, THE Backup_System SHALL perform checksum verification on the backup artifact.
3. WHEN a restore runs in Dry_Run mode, THE Backup_System SHALL perform the Schema_Compatibility_Check between the backup schema version and the target database schema version.
4. WHEN all Dry_Run validation steps complete, THE Backup_System SHALL report a Dry_Run result to the Global_Admin within 60 seconds of validation completion, where the result includes an overall outcome of PASS or FAIL and the individual outcome of each validation step.
5. IF checksum verification fails, THEN THE Backup_System SHALL set the Dry_Run overall outcome to FAIL, SHALL include in the result an indication identifying the checksum mismatch, and SHALL make no change to the target database.
6. IF the Schema_Compatibility_Check detects an incompatibility between the backup schema version and the target database schema version, THEN THE Backup_System SHALL set the Dry_Run overall outcome to FAIL, SHALL include in the result an indication identifying each detected incompatibility, and SHALL make no change to the target database.
7. WHILE a restore is running in Dry_Run mode, THE Backup_System SHALL leave the target database content and schema unchanged from their state immediately before the Dry_Run started.

### Requirement 12: Maintenance Mode and Automatic Rollback for Full Restore

**User Story:** As a Global Admin, I want a full restore to run safely with maintenance mode and automatic rollback, so that a failed restore does not leave the platform in a broken state.

#### Acceptance Criteria

1. WHEN a full restore begins, THE Backup_System SHALL enable Maintenance_Mode within 10 seconds and SHALL keep Maintenance_Mode enabled until the Restore_Job reaches a terminal state of `completed` or `failed`.
2. IF Maintenance_Mode cannot be enabled within 10 seconds of the full restore beginning, THEN THE Backup_System SHALL abort the Restore_Job before applying any backup data, mark the Restore_Job as `failed` with a human-readable reason indicating maintenance mode could not be enabled, and leave the target database unchanged.
3. WHEN Maintenance_Mode is enabled, THE Backup_System SHALL create a Pre_Restore_Snapshot of the current target database state, ordered per the canonical full-restore sequence of criterion 15 (after every Standby_Node has been fenced/detached and before any destructive `--clean` apply), so that no backup data is applied before the Pre_Restore_Snapshot exists and the standby is isolated.
4. IF the Pre_Restore_Snapshot cannot be created, THEN THE Backup_System SHALL abort the Restore_Job before applying any backup data, mark the Restore_Job as `failed` with a human-readable reason indicating snapshot creation failed, leave the target database unchanged, and disable Maintenance_Mode.
5. WHEN the restore data has been applied to the isolated primary in the canonical full-restore sequence of criterion 15, THE Backup_System SHALL run validation checks against the restored database before completing the Restore_Job, where validation checks confirm that: (a) all tables present in the Pre_Restore_Snapshot exist in the restored database, (b) for a full restore performed with `pg_dump` custom-format `--clean` (the target table is dropped/cleaned then repopulated from the backup), each restored table's row count is exactly equal to the count recorded in the applied backup, while for a non-clean additive restore mode each restored table's row count is greater than or equal to the count recorded in the applied backup, and (c) referential integrity constraints report zero violations, where check (c) is a defence-in-depth assertion — PostgreSQL foreign-key enforcement already guarantees referential integrity on a `--clean` restore — and is NOT the primary integrity gate, which is the checksum verification of Requirement 7.4–7.7.
6. WHEN all validation checks pass, THE Backup_System SHALL mark the Restore_Job as `completed`.
7. IF one or more post-restore validation checks fail, THEN THE Backup_System SHALL roll back the isolated primary to the Pre_Restore_Snapshot before any Standby_Node re-seed is attempted (per the canonical full-restore sequence of criterion 15) and mark the Restore_Job as `failed` with a human-readable reason identifying which validation check failed.
8. IF the rollback to the Pre_Restore_Snapshot fails, THEN THE Backup_System SHALL keep Maintenance_Mode enabled, mark the Restore_Job as `failed` with a human-readable reason indicating rollback failed and that manual intervention is required, and SHALL NOT report the platform as available.
9. WHEN the Restore_Job reaches a terminal state of `completed` or after a successful rollback, THE Backup_System SHALL disable Maintenance_Mode within 10 seconds.
10. WHEN a full restore begins on a platform running PostgreSQL Streaming_Replication under CloudNativePG, THE Backup_System SHALL break or suspend streaming replication by detaching or fencing every Standby_Node before taking the Pre_Restore_Snapshot and before applying any backup data to the primary (per the canonical full-restore sequence of criterion 15), so that the restore is never applied to a primary that is still streaming to a live standby.
11. IF streaming replication cannot be broken or suspended (a Standby_Node cannot be detached or fenced) before backup data is applied, THEN THE Backup_System SHALL abort the Restore_Job before applying any backup data, mark the Restore_Job as `failed` with a human-readable reason indicating the standby could not be isolated, leave the target database unchanged, and disable Maintenance_Mode.
12. WHILE a full restore is applying backup data, THE Backup_System SHALL perform the restore only against the isolated primary and SHALL keep every Standby_Node detached or fenced, so that no in-progress restore can corrupt a standby or trigger a failover.
13. WHEN a full restore reaches the terminal state `completed`, THE Backup_System SHALL re-seed or re-create each Standby_Node from the restored primary only after post-restore validation has passed on the isolated primary (per the canonical full-restore sequence of criterion 15) and SHALL resume normal high-availability operation only after each Standby_Node has been re-established, coordinating this transition with the `ha-orchestrated-switchover` spec; the platform SHALL NOT be reported as available with normal HA until the standby has been re-seeded, and the Backup_System SHALL NEVER re-seed a Standby_Node from an unvalidated or rolled-back primary.
14. IF re-seeding or re-creating a Standby_Node from the restored primary fails, THEN THE Backup_System SHALL keep the platform out of normal HA operation, record a human-readable reason indicating standby re-seed failed and that manual intervention is required, and dispatch the configured failure notification, while leaving the restored primary serving traffic.
15. WHEN a full restore runs, THE Backup_System SHALL execute the operation in exactly this canonical ordered sequence: (1) enable Maintenance_Mode; (2) fence/detach every Standby_Node; (3) take the Pre_Restore_Snapshot of the isolated primary; (4) apply the restore with `pg_dump` custom-format `--clean` to the isolated primary; (5) run post-restore validation (criterion 5); (6) ON validation PASS, re-seed each Standby_Node from the restored primary and then resume normal HA; (7) ON validation FAIL, roll back the isolated primary to the Pre_Restore_Snapshot, leave every Standby_Node fenced, do NOT re-seed any Standby_Node, and surface a manual-intervention failure; and THE Backup_System SHALL enforce the invariants that the Pre_Restore_Snapshot MUST be captured and every Standby_Node MUST be fenced BEFORE any destructive `--clean` apply, that a failed-validation rollback MUST restore the Pre_Restore_Snapshot on the isolated primary BEFORE any Standby_Node re-seed is attempted, and that a Standby_Node SHALL NEVER be re-seeded from an unvalidated or rolled-back primary.
16. WHILE a Restore_Job is in a pre-apply phase — that is, before the destructive `pg_restore --clean` apply of step (4) of criterion 15 has begun (covering artifact download, checksum verification, schema-compatibility check, Maintenance_Mode enable, standby fence, and Pre_Restore_Snapshot capture) — THE Backup_System SHALL allow a Global_Admin to cancel the Restore_Job, SHALL stop the restore having applied no backup data to the target database, SHALL release any pre-apply state it established (disable Maintenance_Mode within 10 seconds and, where it had fenced a standby solely for this cancelled restore, restore normal HA), and SHALL record the Restore_Job as `cancelled`.
17. IF a cancel is requested for a Restore_Job after the destructive `pg_restore --clean` apply of step (4) has begun, THEN THE Backup_System SHALL refuse the cancel, return an error response indicating the restore can no longer be safely cancelled, and SHALL allow only the built-in post-restore validation and automatic-rollback path (criteria 5–9, 15) to determine the terminal outcome.

> **Design Decision — Full restore vs live streaming replication (resolves Open Question 5).**
> The platform runs CloudNativePG with at least one live streaming Standby_Node. Restoring a full dump into a primary that is still replicating would stream the destructive `--clean`/reload to the standby (corrupting it) or trip an unintended failover mid-restore. Evaluated options: (i) restore into the live pair and "hope" replication tolerates it — rejected as unsafe; (ii) fail over to the standby first, restore the old primary, then fail back — adds avoidable role churn; (iii) fence/detach the standby, restore the isolated primary, then re-seed the standby from the restored primary and resume HA. **Recommendation:** option (iii). **Rationale:** isolating the primary makes the restore a single-node operation with no replication target to corrupt, and re-seeding the standby afterwards guarantees the standby is a faithful copy of the restored data rather than a partially-streamed intermediate state. This sequence is coordinated with Maintenance_Mode (criteria 1–9) and the existing `ha-orchestrated-switchover` spec, and the full canonical ordering — enable Maintenance_Mode → fence/detach every standby → take Pre_Restore_Snapshot → apply `--clean` restore → validate → on pass re-seed standby and resume HA / on fail roll back the isolated primary before any re-seed — is fixed explicitly in criterion 15.

### Requirement 13: Live Restore Progress and Job Model

**User Story:** As a Global Admin, I want to watch live progress of a backup or restore, so that I know the operation is proceeding and how far along it is.

#### Acceptance Criteria

1. THE Backup_System SHALL represent each backup operation as a Backup_Job and each restore operation as a Restore_Job, each having a status attribute whose value is exactly one of the lifecycle states: queued, running, completed, failed, or cancelled.
2. WHILE a Backup_Job or Restore_Job is in the running state, THE Backup_System SHALL emit, at intervals no greater than 5 seconds apart, either a progress update that increases the percentage value (between 0 and 100 inclusive) by at least 1, OR — WHERE the job is in a long-running monotonic phase that produces no percentage change (for example a multi-gigabyte File_Blob download or a long `pg_restore`) — a liveness heartbeat that asserts the operation is still making progress without requiring a percentage increase.
3. WHEN the Global Admin requests the current state of a Backup_Job or Restore_Job, THE Backup_System SHALL return the job's current status, current progress percentage (0 to 100), the elapsed running time in seconds, and the time since the most recent progress update or liveness heartbeat.
4. WHEN a Backup_Job or Restore_Job transitions to the completed, failed, or cancelled state, THE Backup_System SHALL record its final status, completion timestamp in UTC, and an outcome summary indicating success or the reason for non-success.
5. IF a Backup_Job or Restore_Job produces neither a percentage-increasing progress update NOR a liveness heartbeat for more than 60 consecutive seconds while in the running state, THEN THE Backup_System SHALL transition the job to the failed state and record an outcome summary indicating a progress timeout, such that a long-running monotonic phase that continues to emit liveness heartbeats SHALL NOT be force-failed by this timeout and the timeout fires only on the genuine absence of BOTH a percentage increase AND a heartbeat for the timeout window.
6. IF the Global Admin requests the state of a job identifier that does not correspond to an existing Backup_Job or Restore_Job, THEN THE Backup_System SHALL return a not-found error response and SHALL NOT create a new job.

### Requirement 14: Per-Organisation Restore

**User Story:** As a Global Admin, I want to restore a single organisation's data from a full backup, so that I can recover one organisation without affecting the others.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to select exactly one organisation, identified by its `org_id`, from a Full_Backup for restore.
2. THE Backup_System SHALL obtain the selected organisation's Org_Scoped_Rows from the Full_Backup using the Per-Organisation Data Extraction Mechanism of Requirement 31 (reading the organisation's Per_Org_Logical_Export where one was emitted, otherwise staging the full dump into an ephemeral scratch database and extracting the organisation's rows by `org_id`), based on the Tenant_Isolation_Strategy of row-level `org_id` association enforced by PostgreSQL RLS, rather than treating extraction from the full custom-format `pg_dump` as a trivial or zero-cost operation.
3. WHEN a single-organisation restore runs, THE Backup_System SHALL perform insert, update, and delete operations only on Org_Scoped_Rows whose `org_id` equals the selected organisation's `org_id`, and SHALL NOT insert, update, or delete any Org_Scoped_Row whose `org_id` differs from the selected organisation's `org_id`; this prohibition applies specifically to Org_Scoped_Rows belonging to a different organisation and does not preclude the read-and-ensure-exists handling of Shared_Global_Rows defined in criterion 7.
4. THE Backup_System SHALL allow a Global_Admin to choose the restore granularity as exactly one of: (a) the whole organisation, or (b) a Global_Admin-selected subset of specific entities or tables belonging to that organisation.
5. THE Backup_System SHALL allow a Global_Admin to choose exactly one conflict-handling policy for organisation data that already exists in the target, from: (a) restore as new records, (b) skip existing records, or (c) overwrite existing records.
6. WHEN the conflict-handling policy is restore as new records, THE Backup_System SHALL assign new identifiers to the restored Org_Scoped_Rows and rewrite all references between restored Org_Scoped_Rows so that referential integrity is preserved with zero dangling references, AND where a restored Org_Scoped_Row references a Shared_Global_Row, THE Backup_System SHALL resolve that reference to the ensured-existing Shared_Global_Row in the target (per criterion 7) rather than minting a new shared/global record.
7. WHERE a restored Org_Scoped_Row references another row outside the selected restore scope, THE Backup_System SHALL handle the referenced row according to its classification: WHERE the referenced row is an Org_Scoped_Row of the selected organisation, THE Backup_System SHALL include it transitively unless an equivalent row with the same `org_id` already exists in the target; and WHERE the referenced row is a Shared_Global_Row, THE Backup_System SHALL perform a read-and-ensure-exists — inserting the Shared_Global_Row only if no equivalent row already exists in the target and NEVER modifying or deleting an existing Shared_Global_Row — so that referential integrity holds with zero modification of any other organisation's Org_Scoped_Rows and zero mutation of existing Shared_Global_Rows.
8. IF the selected Full_Backup cannot be read or fails integrity validation, THEN THE Backup_System SHALL abort the restore, make no insert, update, or delete to the target, and return an error indicating the backup is unreadable or corrupt.
9. IF the selected `org_id` is not present in the Full_Backup, THEN THE Backup_System SHALL abort the restore, make no insert, update, or delete to the target, and return an error indicating the selected organisation was not found in the backup.
10. IF any error occurs after restore operations have begun, THEN THE Backup_System SHALL roll back all changes made during that restore so that the target returns to its pre-restore state, and SHALL return an error indicating the restore failed and was rolled back.

> **Design Decision — Org-scoped vs shared/global reference data during per-org restore (resolves Open Question 4).**
> Criterion 3's blanket prohibition on touching any row whose `org_id` differs from the selected org directly contradicted criterion 7's requirement to transitively include referenced rows outside the selected scope, because shared/global platform reference data (subscription plans, global vehicle/CarJam records, feature flags) is exactly what criterion 7 must pull in but criterion 3 forbade. Evaluated options for resolving the contradiction: (i) treat every referenced row as org-scoped and refuse to touch anything cross-org — rejected because it breaks referential integrity for any restored row that depends on shared/global reference data; (ii) allow the restore to freely insert/update shared/global rows — rejected because mutating shared/global rows can corrupt other organisations' data and the platform's reference tables; (iii) **classify rows as Org_Scoped_Row vs Shared_Global_Row and apply different rules per class** — the cross-org prohibition (criterion 3) binds only Org_Scoped_Rows of a different organisation, while shared/global references are handled by a read-and-ensure-exists that inserts a Shared_Global_Row only if absent and never modifies or deletes an existing one (criterion 7). **Recommendation:** option (iii). **Rationale:** a row carrying an `org_id` is tenant data owned by exactly one org and must never be cross-contaminated, whereas a row with no `org_id` (or one in the enumerated global-reference-table set) is platform reference data shared by all orgs; ensure-exists preserves referential integrity for the restored org while guaranteeing zero modification of any other org's tenant data and zero mutation of existing shared/global rows, which is the only resolution that satisfies both criterion 3 and criterion 7 simultaneously. **Residual design detail:** the concrete enumerated set of global-reference tables is to be finalised in design (see Open Question 4).

### Requirement 15: Per-Organisation Backup Browsing

**User Story:** As a Global Admin, I want to browse the contents of a backup for a chosen organisation before restoring, so that I can confirm what will be restored.

#### Acceptance Criteria

1. WHEN a Global_Admin selects a Full_Backup and an organisation, THE Backup_System SHALL load and present the organisation's data grouped by entity type within 5 seconds for browsing before restore, serving the presentation from the precomputed Per_Org_Index emitted at backup time (Requirement 7.9) rather than by staging the full custom-format `pg_dump`, so that the 5-second bound is met without a full-dump stage. (Full per-organisation DATA extraction for an actual restore is handled by the Per-Organisation Data Extraction Mechanism of Requirement 31 — reading a Per_Org_Logical_Export where present, otherwise staging the full dump — and may be heavier than browsing; this SLA covers browsing only.)
2. WHEN the organisation's backup data is presented, THE Backup_System SHALL display, for each entity type, the entity type name and its record count, including entity types that contain zero records.
3. IF the selected Full_Backup cannot be read or is incomplete for the chosen organisation, THEN THE Backup_System SHALL reject the browse request, display an error message indicating the backup could not be loaded, and retain the Global_Admin's backup and organisation selection.
4. WHILE a Global_Admin is browsing the chosen organisation's backup, THE Backup_System SHALL allow the Global_Admin to select or deselect specific entity types or tables for restore.
5. WHEN a Global_Admin has selected at least one entity type or table, THE Backup_System SHALL display a restore-scope summary listing each selected entity type and its record count before the restore is initiated.
6. IF a Global_Admin attempts to initiate the restore with zero entity types or tables selected, THEN THE Backup_System SHALL prevent the restore from starting and display a message indicating that at least one entity type or table must be selected.

> **Design Decision — Per-organisation browse performance (resolves Open Question 3 for browsing).**
> A custom-format `pg_dump` is not partitioned by org, so meeting a 5-second per-org browse SLA by reading the dump directly is infeasible without first staging the whole dump into a scratch database (slow). Evaluated options: (1) stage the full dump into a scratch DB on each browse — simple but far exceeds 5s; (2) parse the custom-format archive directly per browse — fragile and still not partitioned by org; (3) pre-emit a Per_Org_Index at backup time (per-org entity counts/identifiers, optionally a Per_Org_Logical_Export). **Recommendation:** option (3) — emit a Per_Org_Index at backup time (Requirement 7.9) and serve browsing from it. **Rationale:** the per-org summary is computed once, cheaply, while the dump source is already open at backup time, making the 5-second browse SLA achievable without any restore-time staging. Full per-org DATA extraction for an actual restore is a separate, heavier path handled by the Per-Organisation Data Extraction Mechanism of Requirement 31 (resolves Open Question 3).

### Requirement 16: Client-Side Encryption and Key Management

**User Story:** As a Global Admin responsible for customer PII, I want backups encrypted before they leave the platform, so that customer data is unreadable to the third-party cloud provider.

#### Acceptance Criteria

1. WHEN a backup artifact (the `pg_dump` database dump or a File_Blob) is ready for upload, THE Backup_System SHALL encrypt the artifact client-side using the `envelope_encrypt` mechanism in `app/core/encryption` under the Backup_Data_Key hierarchy — a per-backup (or per-key-version) Backup_Data_Key wrapped by the escrowed Backup_Master_Key — and SHALL NOT encrypt backup artifacts under the deployment's `ENCRYPTION_MASTER_KEY`, before transmitting any byte of the artifact to a Cloud_Provider. (Rationale: `ENCRYPTION_MASTER_KEY` is lost in a disaster, so encrypting artifacts under it would make fresh-deployment restore impossible; the escrowed Backup_Master_Key must be able to recover whatever encrypted the artifacts.)
2. IF client-side encryption of a backup artifact fails, THEN THE Backup_System SHALL abort the upload, SHALL NOT transmit any portion of the unencrypted or partially encrypted artifact to the Cloud_Provider, SHALL retain the source data unchanged, and SHALL return an error response indicating that encryption failed.
3. THE Backup_System SHALL store database credentials and secrets contained in a backup only in `envelope_encrypt` encrypted form and SHALL NOT write them in plaintext within the backup artifact.
4. THE Backup_System SHALL exclude database credentials and secrets from all log output in plaintext, substituting a redaction placeholder in their place.
5. WHEN a restore is performed on a fresh deployment, THE Backup_System SHALL obtain the Backup_Data_Key required to decrypt the artifact, identified by the key version recorded in the backup artifact, by unwrapping it with key material supplied independently of the destroyed deployment (an operator-supplied passphrase or exported key that unwraps the stored wrapped Backup_Data_Key, or retrieval from an external key store/KMS), and SHALL decrypt the artifact using that key.
6. THE Backup_System SHALL store the key material needed to decrypt backups (the Backup_Master_Key and any wrapped Backup_Data_Key) recoverably and independently of the deployment being backed up, such that a total loss of that deployment does not lose the ability to decrypt its backups, and SHALL NOT rely solely on key material held only on the deployment being backed up.
7. WHEN a restore is requested on a fresh deployment, THE Backup_System SHALL require the operator to supply the key material (passphrase, exported key, or external KMS reference) needed to unwrap the Backup_Data_Key before any decryption is attempted, and SHALL NOT assume the original deployment's `ENCRYPTION_MASTER_KEY` is present on the fresh deployment.
8. IF the required key material is not available to a fresh deployment when a restore is requested, THEN THE Backup_System SHALL refuse the restore, SHALL NOT write partial or corrupted data to the target, and SHALL return an error response indicating that the decryption key material must be supplied before the restore can proceed.
9. IF, during a restore, the key version recorded in the backup artifact is unavailable from the supplied key material or the obtained key fails to decrypt the artifact (key mismatch), THEN THE Backup_System SHALL abort the restore, SHALL NOT write partial or corrupted data to the target, and SHALL return an error response indicating which key version was required and that the key was unavailable or did not match.
10. WHEN the encryption key is rotated, THE Backup_System SHALL assign a new key version for subsequent backups and SHALL retain all prior key versions for the configured backup retention period so that artifacts encrypted under any prior key version remain restorable until their retention period expires.
11. THE Backup_System SHALL encrypt every backup artifact (the `pg_dump` database dump and every File_Blob) client-side under the Backup_Data_Key hierarchy before writing any byte to ANY Cloud_Provider destination type, including a NAS_Destination (Requirement 29) and an S3_Destination (Requirement 28), so that even a NAS on the local network or an onshore S3-compatible bucket holds only encrypted artifacts and never plaintext customer data.
12. THE Backup_System SHALL expose the current backup-key state of this deployment, reporting at minimum whether an active Backup_Master_Key/Backup_Data_Key is present on this deployment (`has_active_key`), the active key version when present (`active_version`), and whether first-run key setup has completed (`setup_complete`), so that the restore flow can determine whether fresh-deployment key material must be supplied per criterion 7 and the key-management interface can distinguish first-run setup from re-export and rotation. WHEN no active key is present on this deployment, THE Backup_System SHALL report `has_active_key` as false and `setup_complete` according to whether any key version has ever been established.

> **Design Decision — Backup key custody and fresh-deployment bootstrap (resolves Open Questions 1 and 2).**
> Evaluated options for where decryption key material lives and how a fresh deployment obtains it:
> - **(a) Operator-supplied passphrase-wrapped key (escrow/export).** A backup-specific Backup_Master_Key is exported out-of-band; per-backup Backup_Data_Keys are wrapped by it. At restore the operator supplies a passphrase that unwraps the stored Backup_Data_Key. Works on a fresh box with no external service.
> - **(b) External KMS.** Data keys are wrapped by a cloud KMS key (e.g. AWS KMS, GCP KMS). Strong custody and rotation, but introduces an external dependency that may be unavailable on Docker/Colima/Raspberry Pi deployments and during a true DR event.
> - **(c) Key escrow service.** A dedicated escrow holds key material. Adds operational complexity and another system to recover.
>
> **Recommendation:** option (a) — a Backup_Master_Key escrowed/exported out-of-band and re-supplied at restore via an operator passphrase that unwraps the stored Backup_Data_Key — adopted as the portable default, while remaining compatible with (b) an external KMS where one is available (the wrapped Backup_Data_Key model supports either wrapping authority). **Rationale:** option (a) is the only mechanism that works on a fresh box with no external dependency, which is the core DR scenario; the current `ENCRYPTION_MASTER_KEY` must NOT be assumed present on a fresh deployment (it is part of what is lost), so key material must be recoverable independently of the backed-up deployment per criteria 6–8. Rotation (criterion 10) is handled by versioned Backup_Data_Keys retained for the retention period rather than re-encrypting historical backups.

### Requirement 17: Audit Logging

**User Story:** As a Global Admin, I want every backup and restore action recorded, so that there is a complete, attributable history of DR operations.

#### Acceptance Criteria

1. WHEN a Cloud_Provider is connected or disconnected, THE Backup_System SHALL write an Audit_Log entry populating action (`cloud_provider.connected` or `cloud_provider.disconnected`), entity_type (`cloud_provider`), entity_id (the affected Cloud_Provider identifier), user_id (the acting Global_Admin), and created_at (a UTC timestamp with timezone, recorded to at least millisecond precision).
2. WHEN backup or restore configuration is changed, THE Backup_System SHALL write an Audit_Log entry populating action (`backup_config.changed` or `restore_config.changed`), entity_type (`backup_config` or `restore_config`), user_id (the acting Global_Admin), created_at (a UTC timestamp with timezone, to at least millisecond precision), and before_value and after_value capturing the changed configuration fields.
3. WHEN a backup is created or deleted, THE Backup_System SHALL write an Audit_Log entry populating action (`backup.created` or `backup.deleted`), entity_type (`backup`), entity_id (the affected backup identifier), user_id (the acting Global_Admin), and created_at (a UTC timestamp with timezone, to at least millisecond precision).
4. WHEN a restore is triggered and when a restore completes, THE Backup_System SHALL write a separate Audit_Log entry for each event populating action (`restore.triggered` or `restore.completed`), entity_type (`restore`), entity_id (the affected restore identifier), user_id (the acting Global_Admin), and created_at (a UTC timestamp with timezone, to at least millisecond precision).
5. THE Backup_System SHALL exclude secrets, tokens, credentials, and API keys from all Audit_Log fields, including the before_value and after_value fields.
6. THE Backup_System SHALL make backup and restore actions auditable using a durable write-ahead pattern: before an action is applied THE Backup_System SHALL durably record a write-ahead Audit_Log entry capturing the actor, action type, target, and start timestamp, and after the action THE Backup_System SHALL record a completion Audit_Log entry.
7. IF durably recording the write-ahead Audit_Log entry fails before a backup or restore action has begun, THEN THE Backup_System SHALL abort the operation before applying any change and return an error response indicating that the operation could not be audited.
8. IF recording the completion Audit_Log entry fails after a restore or backup has already completed, THEN THE Backup_System SHALL NOT undo or roll back the completed operation, SHALL durably queue the completion audit record for asynchronous retry, and SHALL flag the operation for follow-up so the audit trail is reconciled, so that a transient audit-write failure never reverses an otherwise-successful restore.
9. THE Backup_System SHALL store Audit_Log entries as append-only records and SHALL reject any attempt to update or delete an existing Audit_Log entry.

### Requirement 18: Configurable Notifications

**User Story:** As a Global Admin, I want to be notified about backup and restore outcomes through my chosen channels, so that I learn about failures promptly.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to independently enable or disable notifications for each of the following events: backup failure, backup success, restore failure, and restore success.
2. WHERE email notification is enabled, WHEN a configured notification event occurs, THE Backup_System SHALL dispatch an email notification within 60 seconds of the event using the existing notification pipeline (`app/tasks/notifications.py`, `app/modules/notifications/service.py`).
3. WHERE SMS notification is enabled, WHEN a configured notification event occurs, THE Backup_System SHALL dispatch an SMS notification within 60 seconds of the event via the Connexus SMS integration.
4. WHERE webhook notification is enabled, WHEN a configured notification event occurs, THE Backup_System SHALL dispatch a Webhook_Notification within 60 seconds of the event.
5. THE Backup_System SHALL support any combination of the email, SMS, and webhook notification channels being enabled simultaneously, and SHALL dispatch the notification on each enabled channel independently.
6. WHEN a Backup_Job fails and backup-failure notification is enabled, THE Backup_System SHALL dispatch a notification on every enabled channel within 60 seconds of the failure.
7. WHEN a Backup_Job succeeds and backup-success notification is enabled, THE Backup_System SHALL dispatch a notification on every enabled channel within 60 seconds of the success.

IF dispatch on an enabled channel fails, THEN THE Backup_System SHALL retry that channel up to 3 times, SHALL continue dispatching on all other enabled channels independently of that channel's outcome, and SHALL record a delivery-failure indication for the channel that remains failed after the final retry.

IF an enabled channel has no resolvable recipient (an enabled email channel with neither an explicit address nor any `global_admin` email, or an enabled SMS channel with no explicit number), THEN THE Backup_System SHALL record a delivery-failure indication for that channel rather than reporting a successful dispatch, so that alerts whose recipients resolve to nobody are surfaced rather than silently lost. (The webhook channel is addressed by its configured `webhook_url` per criterion 4 and Requirement 19.)
12. THE Backup_System SHALL provide a Global_Admin-initiated test action that dispatches a test notification on each currently enabled channel (email, SMS, and webhook) to the recipients resolved per criterion 11, and SHALL report the per-channel success or failure of that test dispatch, so that a Global_Admin can verify the alert path before relying on it during an actual disaster; the test action SHALL NOT alter any backup, restore, configuration, or job state.

#### Acceptance Criteria

### Requirement 21: Uploaded File Backup (Content-Addressed)

**User Story:** As a Global Admin, I want backups to include the uploaded files held on the filesystem alongside the database dump, captured incrementally so that unchanged files are not re-uploaded every run, so that a restored deployment has every file referenced by the database present on disk without re-encrypting and re-uploading the entire multi-gigabyte file tree on every backup.

#### Acceptance Criteria

1. WHERE the Backup_Scope is `organisations_only` or `both`, THE Backup_System SHALL determine the candidate Uploaded_Files set by capturing the contents of the `/app/uploads/` and `/app/compliance_files/` storage roots wholesale (every category subfolder, whether currently known or added in future), and SHALL store each captured file as an individually-encrypted, content-addressed File_Blob referenced by the Full_Backup's File_Index.
2. THE Backup_System SHALL NOT use a hardcoded allowlist of upload categories to determine which files are captured, and SHALL exclude database-stored (BYTEA) assets — including organisation branding images on the `platform_branding` table and the legacy `/app/uploads/branding/` disk copy — from the File_Blob set, because those bytes are already captured inside the `pg_dump` database dump (see the Uploaded_Files glossary entry).
3. WHEN capturing a candidate file, THE Backup_System SHALL compute the file's Content_Hash and SHALL upload a File_Blob for that content only when no File_Blob with that content already exists at the destination, so that unchanged or duplicate files are deduplicated and not re-uploaded.
4. WHEN a File_Blob is uploaded, THE Backup_System SHALL encrypt that File_Blob client-side using the `envelope_encrypt` mechanism in `app/core/encryption` under the Backup_Data_Key hierarchy (the per-backup Backup_Data_Key wrapped by the escrowed Backup_Master_Key, per Requirement 16), NOT under the deployment's `ENCRYPTION_MASTER_KEY`, before any byte of the File_Blob is transmitted to the Cloud_Provider, such that no unencrypted File_Blob is ever uploaded and the escrowed key material can decrypt every File_Blob on a fresh deployment.
5. THE Backup_System SHALL name each File_Blob using a keyed/HMAC hash of the file's plaintext content (not the bare plaintext hash) or an otherwise equivalent scheme, so that the Cloud_Provider cannot infer plaintext equality across blobs from blob names while the platform's own File_Index still enables deduplication.
6. WHEN a Full_Backup is produced, THE Backup_System SHALL record in the Backup_Manifest's File_Index, for every captured file, its `file_key` or path, owning `org_id` (or global indicator), Content_Hash, and byte size, and SHALL record the Uploaded_Files count and total Uploaded_Files bytes.
7. WHERE the Backup_Scope is `settings_only`, THE Backup_System SHALL exclude organisation-scoped customer files from the File_Blob set, and SHALL include any organisation-independent or global filesystem-backed uploaded assets that exist (for example global page-editor media).
8. WHEN a Backup_Job runs, THE Backup_System SHALL store the File_Blobs and the File_Index in the same Cloud_Provider location as the database artifact and the Backup_Manifest, as part of one backup artifact set.
9. IF a file enumerated for inclusion cannot be read (permission error, broken symlink, or missing file), THEN THE Backup_System SHALL record the skipped file path in the Backup_Job error description, omit that file from the File_Index, continue capturing the remaining files, and report the count of skipped files in the Backup_Manifest.
10. IF a File_Blob fails to be produced or fails to upload to the Cloud_Provider, THEN THE Backup_System SHALL set the Backup_Job status to `failed`, retain a human-readable error description indicating the file-blob failure, leave any prior successful Full_Backup artifacts and previously stored File_Blobs in the Cloud_Provider unchanged, and dispatch the configured failure notification within 60 seconds of detecting the failure.
11. WHERE the Backup_System emits a Per_Org_Logical_Export for an organisation (per Requirement 31), THE Backup_System SHALL store that Per_Org_Logical_Export as part of the same backup artifact set in the same Cloud_Provider location as the database dump, the File_Blobs, the File_Index, and the Backup_Manifest, SHALL encrypt it client-side under the Backup_Data_Key hierarchy (NOT under `ENCRYPTION_MASTER_KEY`) with its org-identifying contents inside the encrypted envelope (consistent with Requirement 7.8), and SHALL keep the full `pg_dump` database dump present unconditionally regardless of whether any Per_Org_Logical_Export was emitted.

> **Design Decision — Incremental file backup model.**
> The database is dumped in full every backup (it is comparatively small; keeping it a full dump keeps restore simple). The file tree, by contrast, can be multiple gigabytes, so re-encrypting and re-uploading it every scheduled run would wreck backup time, cloud quota, and pruning — files use an incremental, deduplicated model instead. Evaluated options:
> - **(i) Base + increment chain.** A periodic full file archive plus deltas; restore replays the chain. Smaller per-run uploads, but every restore depends on an intact chain — a pruned or corrupt mid-chain increment can break later restores — and pruning is constrained by chain dependencies.
> - **(ii) Content-Addressed Store (CAS).** Each file is an individually-encrypted blob named by a (keyed) hash of its content; each backup records a File_Index mapping {file_key/path, org_id} → {Content_Hash, size}; a blob is uploaded only if absent at the destination (dedup); restore for a chosen backup fetches exactly the blobs its File_Index references.
>
> **Recommendation:** option (ii) CAS with reference-counted pruning. **Rationale vs (i):** every backup is *independently* restorable with no chain replay (a pruned or corrupt blob from another backup cannot break this backup as long as its own referenced blobs exist), identical files dedupe across organisations and across time, and pruning is safe via reference counting (Requirement 8.9 — a blob is deleted only when no retained backup's File_Index references its Content_Hash). This is an architectural change from the former single-bundle File_Archive model.

### Requirement 22: Uploaded File Restore

**User Story:** As a Global Admin, I want a restore to put uploaded files back on the filesystem consistently with the restored database rows, so that after restore every file referenced by the database exists on disk.

#### Acceptance Criteria

1. WHEN a full restore is performed, THE Backup_System SHALL reassemble the file set strictly from the chosen backup's File_Index, fetching each referenced File_Blob and writing it to its corresponding filesystem path under the `/app/uploads/` and `/app/compliance_files/` storage roots, preserving each file's original category subfolder and path.
2. WHEN a per-organisation restore is performed for a selected `org_id`, THE Backup_System SHALL select the file set from the chosen backup's File_Index filtered to the selected organisation — for org-partitioned categories by matching the `{category}/{org_id}/` path segment, and for non-org-partitioned files by the owning `org_id` recorded in the File_Index (resolved from the referencing database row) — and SHALL NOT create, overwrite, or delete any file belonging to a different organisation.
3. WHEN a per-organisation restore is performed for a selected `org_id`, THE Backup_System SHALL restore a non-org-partitioned file (a file whose storage path carries no `org_id` segment) only when a restored row belonging to the selected organisation references that file, using the owning `org_id` recorded in the File_Index to resolve ownership.
4. WHEN the conflict-handling policy for a per-organisation restore is restore as new records, THE Backup_System SHALL rewrite each restored file's stored `file_key` or path so that it is consistent with the rewritten row identifiers, and SHALL write the corresponding File_Blob content to the rewritten path so that the database reference and the on-disk file remain consistent.
5. WHEN file restore for a Restore_Job has completed, THE Backup_System SHALL run a post-restore file-consistency check that verifies, for every restored row carrying a `file_key` or file path reference, whether the referenced file exists on disk at the expected path. Under a true temporal point-in-time backup (Requirement 23 option (A) atomic snapshot or write-through CAS) this check is expected to find no missing references. Under referential-list consistency (Requirement 23 option (C) dump-driven reference-set capture) a small set of files referenced by dumped rows may be legitimately absent because they were deleted between dump-open and file-capture and were recorded as known-skips per Requirement 21.9; THE Backup_System SHALL classify each such recorded known-skip as EXPECTED-AND-ACCEPTABLE rather than a consistency defect.
6. IF the post-restore file-consistency check finds a referenced file missing on disk that is NOT a recorded known-skip from the backup's capture window — that is, a file that SHOULD be present in the File_Index but whose File_Blob is missing at restore — THEN THE Backup_System SHALL record each such missing reference in the Restore_Job outcome summary and SHALL mark the file-consistency check outcome as `failed` (a genuine defect); WHEREAS for a missing reference that corresponds to a recorded known-skip (a file deleted in the capture window under option (C)), THE Backup_System SHALL record it in the Restore_Job summary as an informational known-skip and SHALL NOT mark the file-consistency check outcome as `failed` on its account.
7. IF restoring a file to its target path fails (write error or insufficient space), THEN THE Backup_System SHALL record the failed file path in the Restore_Job error description and SHALL mark the Restore_Job as `failed` with a human-readable reason indicating file restore failed.

### Requirement 23: Point-in-Time Snapshot Consistency

**User Story:** As a Global Admin, I want the database dump and the captured files within a single backup to represent the same logical point in time, so that a restored deployment has no database row referencing a file that was deleted before capture and no captured file orphaned because its database reference is absent.

#### Acceptance Criteria

1. WHEN a Full_Backup is produced, THE Backup_System SHALL provide one of two defined consistency guarantee levels between the database dump and the captured Uploaded_Files: (A) **true temporal point-in-time consistency** — achieved by an atomic filesystem/block snapshot, or by write-through CAS where each file is captured into the content-addressed store at upload/write time — under which the captured file BYTES are exactly consistent with the database rows captured in the same Full_Backup with no torn writes; or (C) **referential-list consistency** — achieved by dump-driven reference-set capture — under which the captured file LIST is referentially consistent with the dumped rows, but a small set of files referenced by dumped rows MAY be legitimately absent because they were deleted between dump-open time and file-capture time (handled as known-skips per Requirement 21.9). THE Backup_System SHALL record in the Backup_Manifest which guarantee level a given Full_Backup was produced under.
2. WHEN the database dump is produced, THE Backup_System SHALL run the `pg_dump` export within a single transaction snapshot using REPEATABLE READ or serializable export-snapshot semantics, so that the database portion of the Full_Backup is internally consistent.
3. WHERE a file is referenced by a row included in the database dump, THE Backup_System SHALL include that file's content as of the consistent point in time in the Full_Backup's File_Index, subject to the unreadable-or-missing-file handling specified in Requirement 21.
4. WHERE the Full_Backup is produced under guarantee level (A) (atomic snapshot or write-through CAS), THE Backup_System SHALL exclude from the Full_Backup any partial or torn file content caused by writes occurring during capture; WHERE the Full_Backup is produced under guarantee level (C) (dump-driven reference-set capture), THE Backup_System SHALL capture each referenced file's bytes as read at capture time and SHALL record as a known-skip (Requirement 21.9) any referenced file that cannot be captured because it was deleted in the dump-open-to-capture window, rather than guaranteeing torn-write-free byte consistency.
5. WHEN the Full_Backup is complete, THE Backup_System SHALL ensure that, for every file reference in the captured database rows, the corresponding File_Blob is present in the Full_Backup's File_Index (subject to the Requirement 21 missing-file handling), and that the File_Index does not reference content for files that correspond to no captured database row.

> **Design Decision — Point-in-time consistency mechanism (resolves Open Question 14).**
> Neither dump-then-copy nor copy-then-dump file ordering is safe, so ordering is rejected in favour of a defined consistency guarantee level. The two guarantee levels of Requirement 23.1 — (A) true temporal point-in-time (byte consistency) and (C) referential-list consistency (list consistency, with benign expected skips) — are deliberately distinct: only (A) or write-through CAS gives true temporal byte consistency; (C) guarantees the file LIST is referentially consistent but a file deleted between dump-open time T and capture time T+δ yields a dumped row referencing a file that cannot be captured (a legitimate, expected known-skip per Requirement 21.9). Evaluated options:
> - **(A) Atomic filesystem/block snapshot (LVM or ZFS)** of the standby's replicated `/app/uploads/` and `/app/compliance_files/` volumes, taken at the instant the `pg_dump` export snapshot is opened; both sides read as-of that snapshot. Shortest quiesce, true temporal point-in-time, but requires host filesystem support.
> - **(B) Pause the rsync volume sync** (`app/modules/ha/volume_sync_service.py`) so the standby file copy is quiesced/static, open the `pg_dump` export snapshot, capture files, then resume sync. A coordination bridge, not a true instant snapshot.
> - **(C) Dump-driven reference-set capture:** drive file capture from the set of `file_key`s present in the dumped rows, so by construction every captured file matches a dumped row and every dumped row's file is sought (missing files handled per Requirement 21). This gives referential consistency of the LIST, not temporal consistency of the BYTES — a file deleted in the capture window is a legitimate known-skip, not a defect.
> - **(D) Write-through CAS:** capture each file into the content-addressed store at upload/write time, so a backup merely references already-captured content as-of the dumped row. Because the content existed in the CAS at the moment its row was written, the backup is temporally consistent without any filesystem snapshot.
>
> **Recommendation:** a layered approach — prefer **(A)** atomic block snapshot WHERE the host filesystem supports it (shortest quiesce, true point-in-time); adopt **(D) write-through CAS** as the recommended *portable* path to TRUE temporal point-in-time consistency on Docker/Colima/Raspberry Pi where LVM/ZFS snapshots are unavailable; **fall back to (C)** dump-driven reference-set capture (referential-list consistency, with benign known-skips) where write-through CAS is not yet in place; use **(B)** only as a coordination bridge when needed. **Rationale:** (D) achieves real temporal consistency without special filesystem features by moving capture to write time — each file is content-addressed into the store as it is uploaded, so the backup references content that demonstrably existed when its referencing row was written, eliminating the capture-window deletion gap that makes (C) only list-consistent. (C) remains a sound, simple portable default whose only weakness is the benign, expected known-skip; under (C) the post-restore file-consistency check (Requirement 22.5/22.6) treats a capture-window known-skip as informational, not a hard `failed` defect. **Backup-window impact:** option (A)'s quiesce is near-instant; option (D) moves cost to upload time (negligible per file); option (C) adds the cost of enumerating referenced `file_key`s from the dump; none touch the primary, because file reads come from the standby per `byo-drive-backup` and Open Question 13.

### Requirement 24: File Restore Set Reassembly

**User Story:** As a Global Admin, I want a restore to reassemble exactly the file set belonging to the chosen backup, so that the files placed on disk match the database rows being restored from that same backup.

#### Acceptance Criteria

1. WHEN a restore is performed for a chosen Full_Backup, THE Backup_System SHALL determine the file set to reassemble strictly from that backup's File_Index, and SHALL NOT include files referenced only by any other backup's File_Index.
2. WHEN a full restore is performed, THE Backup_System SHALL reassemble the file set from every entry in the chosen backup's File_Index, fetching each entry's File_Blob by its Content_Hash.
3. WHEN a per-organisation restore is performed for a selected `org_id`, THE Backup_System SHALL reassemble the file set from only the chosen backup's File_Index entries whose owning `org_id` equals the selected organisation.
4. IF a File_Blob referenced by the chosen backup's File_Index cannot be fetched or fails its own integrity check, THEN THE Backup_System SHALL record the affected file reference in the Restore_Job error description and SHALL mark the Restore_Job as `failed` with a human-readable reason indicating a referenced File_Blob could not be retrieved.

### Requirement 25: Recovery Objectives (RPO/RTO)

**User Story:** As a Global Admin responsible for DR/BCP, I want explicit recovery-point and recovery-time objectives, so that backup frequency and retention are derived from agreed business continuity targets rather than guessed.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to configure a Recovery_Point_Objective and a Recovery_Time_Objective, each expressed as a duration, and SHALL apply documented sensible defaults (a default Recovery_Point_Objective of 24 hours and a default Recovery_Time_Objective of 4 hours) until a Global_Admin changes them.
2. WHEN a Global_Admin configures the backup frequency or Retention_Policy, THE Backup_System SHALL evaluate whether the interval between successive scheduled backups does not exceed the configured Recovery_Point_Objective, and IF the interval exceeds the Recovery_Point_Objective, THEN THE Backup_System SHALL present a warning identifying that the configured schedule does not satisfy the Recovery_Point_Objective before the configuration is saved.
3. THE Backup_System SHALL record the configured Recovery_Point_Objective and Recovery_Time_Objective values and the timestamp at which each was last changed, and SHALL surface both objectives on the backup configuration view.
4. WHEN a Restore_Rehearsal (Requirement 26) completes, THE Backup_System SHALL record the rehearsal's measured restore duration so it can be compared against the configured Recovery_Time_Objective.
5. IF a completed Restore_Rehearsal's measured restore duration exceeds the configured Recovery_Time_Objective, THEN THE Backup_System SHALL record the Recovery_Time_Objective as unmet for that rehearsal and dispatch a notification on every enabled channel.

> **Design Decision — Concrete RPO/RTO target values require business sign-off.**
> The defaults (24h RPO, 4h RTO) are documented engineering placeholders so the system has safe behaviour out of the box; the concrete contractual target values are a business continuity decision and need explicit business sign-off (see Open Question 17). The system therefore makes the objectives Global-Admin-configurable and validates schedule/retention against them, rather than hard-coding a single target.

### Requirement 26: Scheduled Restore Rehearsals

**User Story:** As a Global Admin, I want automated scheduled restore rehearsals into an isolated scratch environment, so that I have ongoing proof that a backup actually restores, beyond the checksum/schema validation a Dry_Run provides.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to configure a schedule for automated Restore_Rehearsals.
2. WHEN a scheduled Restore_Rehearsal runs, THE Backup_System SHALL restore a recent Full_Backup into an isolated scratch environment that is separate from every production database and Standby_Node, such that the rehearsal applies no change to any production or standby database.
3. WHEN a Restore_Rehearsal has applied the backup into the scratch environment, THE Backup_System SHALL run validation comprising a schema check, a per-table row-count check, a file-consistency check (per Requirement 22.5/22.6 semantics), and an application smoke check against the scratch environment.
4. WHEN a Restore_Rehearsal's validation completes, THE Backup_System SHALL record a rehearsal result of `passed` or `failed` together with the identifier of the rehearsed Full_Backup, the individual outcome of each validation step, and the measured restore duration.
5. WHEN a Restore_Rehearsal has produced its result, THE Backup_System SHALL tear down the isolated scratch environment, releasing its resources, regardless of whether the rehearsal passed or failed.
6. IF a Restore_Rehearsal result is `failed`, THEN THE Backup_System SHALL dispatch a notification on every enabled channel (per Requirement 18) reporting the rehearsal failure and which validation step failed.
7. IF tearing down the scratch environment fails, THEN THE Backup_System SHALL record the scratch-environment identifier and a human-readable teardown-failure reason and dispatch a notification on every enabled channel so the leftover environment can be reclaimed manually.

### Requirement 27: Optional Immutable Second Copy

**User Story:** As a Global Admin guarding against ransomware and credential compromise, I want an optional immutable second copy of backups, so that a compromised admin token or malware cannot delete every backup via the retention or prune path.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to optionally configure a second backup destination as an Immutable_Copy destination that provides delete-protection / object-lock / write-once-read-many (WORM) immutability for a configurable lock window.
2. WHERE an Immutable_Copy destination is configured, WHEN a Full_Backup is produced, THE Backup_System SHALL write a copy of that Full_Backup's encrypted artifacts to the Immutable_Copy destination under its object-lock for the configured lock window.
3. WHILE a backup copy on the Immutable_Copy destination is within its lock window, THE Backup_System SHALL NOT delete or overwrite that copy through any retention, count-limit prune, or garbage-collection path, and the Immutable_Copy destination SHALL reject any such delete or overwrite request.
4. WHERE an Immutable_Copy destination is configured, THE Backup_System SHALL apply retention and prune (Requirement 8) to the Immutable_Copy destination only for copies whose lock window has expired, and SHALL NOT shorten or remove an active lock window.
5. IF writing a Full_Backup copy to the configured Immutable_Copy destination fails, THEN THE Backup_System SHALL record a human-readable error indicating the immutable copy failed and dispatch the configured failure notification, while leaving the primary Cloud_Provider copy unaffected.
6. THE Backup_System SHALL recognise an S3_Destination (Requirement 28) configured with S3 Object Lock in compliance mode as the recommended Immutable_Copy destination for criteria 1–5, because compliance-mode Object Lock provides the write-once-read-many delete-protection this requirement requires for the configured lock window; THE Backup_System SHALL NOT treat a NAS_Destination (Requirement 29) as a substitute for the Immutable_Copy unless that NAS itself provides WORM or immutable-snapshot capability, because a standard NAS share offers no true write-once-read-many protection.

> **Design Decision — Immutable destination choice (recommendation with rationale).**
> Google Drive and OneDrive provide no object-lock/WORM, so a compromised Global_Admin token or ransomware reaching the retention/prune path could delete every backup. Evaluated options: (i) rely solely on Google Drive/OneDrive versioning/trash — rejected because trash and versions are deletable by the same compromised credential; (ii) a second consumer-cloud account — rejected for the same reason (no true lock); (iii) an object-storage destination with native object-lock/WORM (for example Amazon S3 Object Lock in compliance mode, Backblaze B2 Object Lock, or Azure Blob immutability policies) added behind the Storage_Interface. **Recommendation:** option (iii) — an S3-compatible destination with Object Lock in compliance mode for the configured lock window — because compliance-mode object-lock cannot be shortened or bypassed even by the account root, which is exactly the ransomware/insider threat model this requirement addresses. The Immutable_Copy is optional so deployments without such a destination are unaffected, but it is strongly recommended for production DR.

### Requirement 28: S3 / S3-Compatible Destination Configuration and Authentication

**User Story:** As a Global Admin, I want to configure an S3 or S3-compatible bucket as a backup destination authenticated by access keys, so that I can store encrypted backups in AWS S3 or in a self-hosted/onshore S3-compatible service.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to configure an S3_Destination independently from Global Admin settings, consistent with Requirement 2.1, such that configuring or removing the S3_Destination has no effect on the credentials or connection state of any other configured destination.
2. WHEN a Global_Admin configures an S3_Destination, THE Backup_System SHALL accept an access key ID, a secret access key, an optional session token, a bucket name, a region, an optional endpoint URL for S3-compatible providers, and an optional addressing style of exactly one of `path_style` or `virtual_hosted`.
3. IF the bucket name, the access key ID, or the secret access key is absent or empty when an S3_Destination is configured, THEN THE Backup_System SHALL reject the configuration, persist no partial credentials, and return an error response identifying the missing field.
4. THE Backup_System SHALL store the S3_Destination access key ID, secret access key, and session token envelope-encrypted using the existing `app/core/encryption` helpers with the `ENCRYPTION_MASTER_KEY` (operational secrets, the same protection as OAuth tokens in Requirement 2.4 — NOT the escrowed Backup_Data_Key hierarchy used for backup artifacts per Requirement 16), and SHALL never persist these credentials in unencrypted form.
5. WHERE an endpoint URL is configured, THE Backup_System SHALL direct all S3 operations to that endpoint (supporting S3-compatible providers such as MinIO, Backblaze B2, and Wasabi), and WHERE no endpoint URL is configured, THE Backup_System SHALL use the AWS S3 endpoint for the configured region.
6. WHERE an addressing style is configured, THE Backup_System SHALL use path-style or virtual-hosted-style request addressing as configured, and WHERE no addressing style is configured, THE Backup_System SHALL default to virtual-hosted-style addressing.
7. WHEN a Global_Admin saves an S3_Destination configuration, THE Backup_System SHALL perform a connection/reachability test (for example a head-bucket request or a put-then-delete of a test object) before setting the destination state to `connected`, and IF the test fails THEN THE Backup_System SHALL leave the destination `disconnected` and return a human-readable error describing the failure (for example bucket not found, access denied, endpoint unreachable, or region mismatch).
8. THE Backup_System SHALL exclude the S3_Destination access key ID, secret access key, and session token from all log output, including error and debug logs.
9. THE Backup_System SHALL access the S3_Destination exclusively through the Storage_Interface (Requirement 3), SHALL upload large artifacts using S3 multipart upload with the resume and retry semantics of Requirement 4.8, and SHALL encrypt every artifact client-side under the Backup_Data_Key hierarchy before upload (Requirements 16.11 and 21.4).

### Requirement 29: NAS Destination Configuration and Authentication

**User Story:** As a Global Admin, I want to configure a NAS network share or mounted volume as a backup destination, so that I can keep encrypted backups on storage I control, optionally onshore in New Zealand.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to configure a NAS_Destination independently from Global Admin settings, consistent with Requirement 2.1, such that configuring or removing the NAS_Destination has no effect on the credentials or connection state of any other configured destination.
2. WHEN a Global_Admin configures a NAS_Destination, THE Backup_System SHALL accept a network share path or mount target, a share access mode of exactly one of `smb` (SMB/CIFS), `nfs`, or a pre-mounted `volume_path`, a target directory on the share for backup artifacts, and any credentials required by the selected access mode (for example an SMB username and password).
3. IF the network share path / mount target is absent or empty, or a credential required by the selected access mode is missing, THEN THE Backup_System SHALL reject the configuration, persist no partial credentials, and return an error response identifying the missing field.
4. THE Backup_System SHALL store NAS_Destination credentials (for example the SMB username and password) envelope-encrypted using the existing `app/core/encryption` helpers with the `ENCRYPTION_MASTER_KEY` (operational secrets, the same protection as OAuth tokens in Requirement 2.4 — NOT the escrowed Backup_Data_Key hierarchy used for backup artifacts per Requirement 16), and SHALL never persist these credentials in unencrypted form.
5. WHEN a Global_Admin saves a NAS_Destination configuration, THE Backup_System SHALL perform a connection/reachability test that mounts or reaches the share and writes then deletes a test artifact in the configured directory before setting the destination state to `connected`, and IF the test fails THEN THE Backup_System SHALL leave the destination `disconnected` and return a human-readable error describing the failure (for example share unreachable, mount failed, authentication rejected, or directory not writable).
6. WHEN the Backup_System writes a backup artifact to a NAS_Destination, THE Backup_System SHALL write the encrypted artifact into the configured directory on the share using the atomic temp-file-then-rename approach of Requirement 4.9.
7. THE Backup_System SHALL exclude NAS_Destination credentials from all log output, including error and debug logs.
8. THE Backup_System SHALL access the NAS_Destination exclusively through the Storage_Interface (Requirement 3) and SHALL encrypt every artifact client-side under the Backup_Data_Key hierarchy before writing it to the share (Requirements 16.11 and 21.4), so that the share holds only encrypted artifacts.

### Requirement 30: Multiple Simultaneous Backup Destinations

**User Story:** As a Global Admin, I want to configure more than one backup destination at once with one designated primary, so that I can keep a primary cloud copy, an onshore NAS copy, and an immutable S3 second copy together.

#### Acceptance Criteria

1. THE Backup_System SHALL allow a Global_Admin to configure more than one Cloud_Provider destination simultaneously (for example a primary `google_drive`, a secondary `nas`, and an S3 Immutable_Copy), spanning any combination of the supported destination types.
2. THE Backup_System SHALL require exactly one configured destination to be designated the primary destination, and SHALL allow zero or more additional destinations to be designated as copy destinations.
3. WHEN a Backup_Job produces a Full_Backup, THE Backup_System SHALL write the encrypted backup artifact set to every configured destination (the primary destination and each copy destination), consistent with the Immutable_Copy handling of Requirement 27.
4. WHEN a restore is initiated, THE Backup_System SHALL allow the Global_Admin to source the chosen backup from any configured destination that holds that backup, and SHALL restore identically regardless of which destination supplies the artifacts (per Requirement 3).
5. IF writing the Full_Backup to a copy destination fails while the write to the primary destination succeeds, THEN THE Backup_System SHALL record a human-readable error identifying the failed copy destination and dispatch the configured failure notification, while retaining the successful primary copy (consistent with Requirement 27.5).
6. IF writing the Full_Backup to the primary destination fails, THEN THE Backup_System SHALL mark the Backup_Job as `failed` per Requirement 5.7 and dispatch the configured failure notification.
7. THE Backup_System SHALL allow a Global_Admin to edit an existing destination's configuration and to re-designate which configured destination is the primary, and SHALL enforce the exactly-one-primary invariant (criterion 2) transactionally on every such change so that at all times exactly one configured destination is the primary; WHEN a Global_Admin designates a different destination as primary, THE Backup_System SHALL clear the previous primary designation and set the new one in the same atomic transaction, such that the change either fully applies or leaves the prior designation intact, and SHALL refuse any change that would leave zero or more than one primary.

> **Design Decision — Multiple destinations with one primary and optional copy destinations.**
> The resilience and residency model already implies more than one place to put a backup: Requirement 27 wants an immutable second copy, and the new onshore NAS / NZ-region S3 options (Requirements 20, 28, 29) address data residency. Rather than restrict the platform to a single destination, the recommended model allows multiple configured destinations with exactly one designated primary and zero or more copy destinations (one of which may be the Immutable_Copy). **Rationale:** backup writes fan out to every configured destination so a primary cloud drive, an onshore NAS, and an immutable S3 bucket can coexist; restore can source from any destination that holds the chosen backup because all destinations carry the identical encrypted, content-addressed artifact set behind the Storage_Interface (Requirement 3). A copy-destination write failure is surfaced and notified but does not fail the backup as long as the primary write succeeds, whereas a primary write failure fails the Backup_Job.

### Requirement 31: Per-Organisation Data Extraction Mechanism

**User Story:** As a Global Admin, I want a defined mechanism for extracting a single organisation's data from a full backup, so that a per-organisation restore is always possible and is fast in the common case without bloating every backup.

#### Acceptance Criteria

1. WHERE the backup size/time budget allows, WHEN a Full_Backup is produced with Backup_Scope `organisations_only` or `both`, THE Backup_System SHALL emit a Per_Org_Logical_Export (the organisation's Org_Scoped_Rows as a `COPY`/SQL representation keyed by `org_id`) alongside the full `pg_dump` dump for one or more contained organisations, encrypt it client-side under the Backup_Data_Key hierarchy with its org-identifying contents inside the encrypted envelope (consistent with Requirement 7.8), and record its presence for that organisation in the Backup_Manifest and Per_Org_Index (per Requirements 5.9 and 7.9).
2. THE Backup_System SHALL produce the full `pg_dump` custom-format dump in every Full_Backup unconditionally and SHALL treat the full dump as the system-of-record for full restore, such that emitting or not emitting any Per_Org_Logical_Export never reduces or replaces the full dump.
3. WHEN a per-organisation restore is requested for a selected `org_id` and a Per_Org_Logical_Export exists for that organisation in the chosen Full_Backup, THE Backup_System SHALL extract that organisation's data from the Per_Org_Logical_Export rather than staging the full dump.
4. WHEN a per-organisation restore is requested for a selected `org_id` and no Per_Org_Logical_Export exists for that organisation in the chosen Full_Backup, THEN THE Backup_System SHALL stage the full `pg_dump` dump into an ephemeral scratch database and extract the selected organisation's Org_Scoped_Rows from that scratch database by `org_id`.
5. WHEN a per-organisation extraction has staged the full dump into an ephemeral scratch database, THE Backup_System SHALL tear down the ephemeral scratch database after the extraction completes, regardless of whether the extraction succeeded or failed, releasing its resources.
6. THE Backup_System SHALL apply the org-scoped/shared-global classification and conflict-handling rules of Requirement 14 to the extracted rows identically regardless of whether the extraction used a Per_Org_Logical_Export (criterion 3) or staged the full dump (criterion 4), so that the extraction mechanism does not affect the restore outcome.
7. IF a Per_Org_Logical_Export recorded as present for the selected organisation cannot be read or fails its integrity check at restore time, THEN THE Backup_System SHALL fall back to staging the full dump (criterion 4) for that organisation rather than failing the restore.

> **Design Decision — Per-organisation data-extraction mechanism (resolves Open Question 3).**
> Per-organisation browsing (Requirement 15) is already served from the Per_Org_Index, but per-organisation DATA extraction for an actual restore was only deferred — and the choice affects the BACKUP artifact (Requirements 5, 7, 21), not just restore, so it must be committed in the acceptance criteria. Evaluated options:
> - **(a) Stage-and-extract.** Restore the full dump into an ephemeral scratch database, then `COPY`/`SELECT` the selected org's rows out by `org_id` and apply them to the target. Simple and requires no backup-format change, but every per-org restore pays the full-DB staging disk and time cost.
> - **(b) Per-org logical export at backup time.** Emit a per-org logical export (per-org `COPY`/SQL) alongside the full dump so a per-org restore reads only that org's export. Fast restore, but enlarges every backup and adds backup-time cost on every run.
> - **(c) Hybrid.** The full dump remains the system-of-record for full restore; emit a lightweight Per_Org_Logical_Export only WHERE the backup size/time budget allows, falling back to (a) staging when the per-org export is absent.
>
> **Recommendation:** option (c) hybrid. **Rationale:** the full dump is required for full restore regardless, so it stays the system-of-record; emitting a Per_Org_Logical_Export opportunistically gives fast, reliable per-org restore in the common case without forcing backup bloat on every run; and stage-and-extract remains the guaranteed fallback so a per-org restore is always possible even when the per-org export was not emitted. This is the only option that does not force a permanent tradeoff onto either every backup (option b) or every restore (option a). **Residual design detail:** the concrete size/time budget threshold that gates per-org export emission, and the provisioning/teardown of the ephemeral scratch database, are to be finalised in design (see Open Question 3).

## Open Questions and Risks

The following items involve non-trivial design decisions that should be resolved during the design phase rather than assumed here.

1. **Encryption key custody on a fresh deployment. — RESOLVED (now firm requirements; see Requirement 16, criteria 5–9, and the Backup key custody Design Decision).** Key material needed to decrypt backups (Backup_Master_Key and wrapped Backup_Data_Key) MUST be stored recoverably and independently of the deployment being backed up; a fresh deployment MUST obtain it before any decryption via an operator-supplied passphrase/exported key (or external KMS). The current `ENCRYPTION_MASTER_KEY` is not assumed present on a fresh deployment. **Residual design detail:** the concrete escrow/export workflow, passphrase strength rules, and optional KMS integration are to be finalised in design.

2. **Encryption key rotation strategy. — RESOLVED (now a firm requirement; see Requirement 16.10).** Rotation uses versioned Backup_Data_Keys retained for the configured retention period so artifacts under any prior key version remain restorable; historical backups are not re-encrypted. **Residual design detail:** the KEK-wrapping format and the storage layout for retained key versions are to be finalised in design.

3. **Tenant isolation assumption. — RESOLVED for the per-org extraction mechanism (now a firm requirement; see Requirement 31, criteria 1–7, Requirements 5.9, 7.9, 14.2, and the Per-organisation data-extraction Design Decision).** This document assumes row-level `org_id` isolation with RLS (the current OraInvoice model). Per-organisation **browsing** is served from the precomputed Per_Org_Index emitted at backup time (Requirements 7.9 and 15.1, and the per-org browse Design Decision), so browsing does NOT require staging the full dump. Per-organisation **data extraction for an actual restore** is now committed to the hybrid mechanism of Requirement 31: the full `pg_dump` dump remains the unconditional system-of-record; WHERE the backup size/time budget allows, a Per_Org_Logical_Export is emitted per org so per-org restore reads only that org's export; and WHERE no Per_Org_Logical_Export exists for the selected org, the system falls back to staging the full dump into an ephemeral scratch database and extracting the org's rows by `org_id` (tearing the scratch database down afterwards). **Residual design detail:** the concrete size/time budget threshold that gates per-org export emission, and the provisioning/teardown of the ephemeral scratch database, are to be finalised in design.

4. **Per-org restore and cross-org foreign keys. — RESOLVED (now firm requirements; see Requirement 14, criteria 3, 6, and 7, the Org_Scoped_Row and Shared_Global_Row glossary entries, and the org-scoped-vs-shared-global Design Decision under Requirement 14).** The former contradiction between criterion 14.3 (forbidding any touch of a different org's rows) and criterion 14.7 (requiring transitive inclusion of referenced rows outside scope) is resolved by classifying rows as Org_Scoped_Row (carries an `org_id`) versus Shared_Global_Row (no `org_id`, or in the enumerated global-reference-table set). The cross-org prohibition (14.3) binds only Org_Scoped_Rows of a different organisation; shared/global references are handled by a read-and-ensure-exists (14.7) that inserts a Shared_Global_Row only if absent and never modifies or deletes an existing one, with reference rewriting (14.6) resolving to the ensured-existing shared/global row rather than minting a new one. **Residual design detail:** the concrete enumerated set of global-reference tables (for example subscription plans, global vehicle/CarJam records, feature flags) is to be finalised in design.

5. **Maintenance mode and HA streaming-replication handling. — LARGELY RESOLVED (now firm requirements; see Requirement 12, criteria 10–15, and the full-restore-vs-streaming-replication Design Decision).** A full restore MUST break/suspend CloudNativePG streaming replication by detaching/fencing every Standby_Node before applying backup data, restore against the isolated primary, then re-seed/re-create the standby from the restored primary before resuming normal HA — never restoring into a live replicated pair — coordinated with Maintenance_Mode and the `ha-orchestrated-switchover` spec. **Residual design detail:** the design must still specify the concrete mechanism for enforcing Maintenance_Mode platform-wide (middleware flag, load-balancer/nginx toggle, or application setting), how in-flight requests are drained, and the exact CloudNativePG fence/detach and re-seed commands used.

6. **Pre-restore snapshot and rollback mechanism for full restore.** Requirement 12 requires automatic rollback. The design must decide whether the Pre_Restore_Snapshot is a `pg_dump`, a filesystem/volume snapshot, or a restore into a parallel database with an atomic switchover. This interacts with the existing HA topology and the `ha-orchestrated-switchover` spec.

7. **Full-dump size, runtime, and worker placement.** A full `pg_dump` of a 132+ table multi-org database may be large and slow. The design must decide where the dump runs (primary vs standby replica, consistent with `byo-drive-backup` sourcing from standby), concurrency with live traffic, and whether the in-process scheduler in `app/tasks/scheduled.py` is sufficient or a dedicated worker is required.

8. **Restore onto a non-empty target.** A full restore onto a fresh deployment is well-defined, but the behaviour of a full restore onto an already-populated deployment (drop-and-recreate vs refuse) must be specified, especially given that pg_dump custom format supports `--clean`.

9. **Provider API quotas and large-file limits.** Google Drive and OneDrive have differing file-size limits, upload session lifetimes, and rate limits. With the content-addressed model (Requirements 21 and 24), the file portion of a backup is now uploaded as many individually-encrypted per-file File_Blobs rather than one multi-gigabyte bundle, which largely removes the single-large-file concern for files; the design must instead confirm per-request and per-day API rate/quota limits are not exceeded by uploading many small File_Blobs (and consider batching or packing very small blobs). The database dump remains a single artifact, so the multi-GB single-file limit/splitting concern still applies to the dump.

10. **Webhook delivery reliability and security.** The design should decide whether webhook deliveries are signed (HMAC), retried on failure, and how delivery failures are surfaced, to avoid silent loss of monitoring events.

11. **Interaction with `byo-drive-backup` OAuth app registration.** Both features use Google OAuth. The design should confirm whether they share a Google Cloud project/client registration and OAuth scopes, or use distinct registrations, to avoid scope or consent-screen conflicts.

12. **Connexus SMS cost and verbosity.** SMS notifications incur per-message cost. The design should decide whether SMS is restricted to failures and escalations only, to avoid unnecessary spend on success notifications.

13. **Uploaded-file source location relative to the pg_dump.** The database dump and the File_Blobs may be produced on different hosts. Consistent with `byo-drive-backup`, which reads rsync-replicated files on the standby, the design must decide whether the File_Blobs are read from the primary host's volumes or from the standby's rsync-replicated copy (populated by `app/modules/ha/volume_sync_service.py`), and how to ensure the file set is sourced from the same node the dump represents. This interacts with the point-in-time consistency guarantee of Requirement 23. **Risk:** reading files from a standby whose volume sync lags the primary could omit recently uploaded files or include files for rows not present in the dump; the dump-driven reference-set capture of Requirement 23 mitigates this by sourcing the capture list from the dumped rows.

14. **Files orphaned or added between the DB snapshot and the file snapshot. — RESOLVED (now a firm requirement; see Requirement 23 and its Point-in-time consistency Design Decision).** Backup ordering (dump-then-copy / copy-then-dump) is rejected as unsafe; instead the dump and the file capture MUST represent one consistent logical point in time. The recommended mechanism is an atomic block snapshot where the host filesystem supports it, falling back to dump-driven reference-set capture as the portable default. The post-restore file-consistency check (Requirement 22.5) is therefore expected to pass without dangling references. **Residual design detail:** detection of host snapshot capability and the exact quiesce/resume coordination with `volume_sync_service.py` are to be finalised in design.

15. **Non-org-partitioned upload paths during per-organisation restore. — LARGELY RESOLVED for branding (see the Uploaded_Files glossary entry and Requirement 21.2).** Organisation branding images are authoritatively stored as BYTEA on `platform_branding` and are captured inside the `pg_dump` dump, so branding org-ownership travels with its database row and branding is NOT part of the File_Blob set — this removes branding from the non-org-partitioned file-isolation problem. **Remaining concern:** any other org-independent filesystem-backed media (for example global page-editor media) is still resolved to an owning `org_id` via the referencing database row and recorded in the File_Index (Requirements 21.6 and 22.3); the design must confirm that such global assets are handled without leaking one organisation's media into another's per-org restore.

16. **Payroll and HR PII categories are in scope, including biometric data.** Payslip PDFs (`payslips`) and staff documents and photos (`attachments`, `staff_photos`, `clock_photos`) are payroll/HR personally identifiable information and are included in the Uploaded_Files set. In particular, kiosk clock-in photos (`clock_photos`) are likely **biometric information under the NZ Privacy Act 2020**, are in scope for backup, are encrypted client-side, and will leave to a US-based cloud provider; this heightened-sensitivity category is surfaced for explicit Global_Admin acknowledgement in the data-residency notice (Requirement 20.6). These categories must be encrypted like every other backup artifact; this is already covered by the client-side `envelope_encrypt` mechanism (Requirements 16 and 21.4), but the design should confirm these sensitive categories are explicitly in scope for capture, encryption, retention, and per-org isolation.
17. **Concrete RPO/RTO target values need business sign-off.** Requirement 25 makes the Recovery_Point_Objective and Recovery_Time_Objective Global-Admin-configurable with documented engineering defaults (24h RPO, 4h RTO), and validates backup frequency/retention against them. The concrete contractual target values, however, are a business continuity decision: the design/rollout must obtain explicit business sign-off on the agreed RPO/RTO before the defaults are treated as the platform's committed objectives, and confirm that the scheduled backup frequency, retention, and rehearsal cadence (Requirements 8, 25, 26) satisfy the signed-off targets.

18. **Immutable second-copy destination selection.** Requirement 27 makes an Immutable_Copy destination optional and recommends an S3-compatible Object Lock (compliance mode) target behind the Storage_Interface, because Google Drive/OneDrive offer no WORM/object-lock. The design must confirm the chosen immutable destination, its lock-window duration relative to the Retention_Policy, the credentials model (ideally write-only/append credentials that cannot themselves delete locked objects), and how the Immutable_Copy interacts with the per-blob content-addressed model (whether the immutable copy stores the full artifact set or only the dump).

19. **NAS availability and mount reliability during scheduled backups.** Requirement 29 adds a NAS_Destination reached as an SMB/CIFS or NFS share or a mounted volume. The design must decide how the Backup_System behaves when the NAS share is unreachable or unmounted at scheduled backup time (retry-with-backoff, skip-with-alert, or fail the Backup_Job), how mount/reachability state is verified before each scheduled run (reusing the Requirement 29.5 connection test), and how transient LAN/NAS outages interact with the Backup_Window (Requirement 8) and the multiple-destination fan-out (Requirement 30). **Risk:** a NAS that is offline or whose mount has gone stale at run time could silently miss recovery points unless reachability is verified and surfaced as a failure notification.

20. **S3 endpoint/region validation and S3-compatible behaviour.** Requirement 28 supports AWS S3 and S3-compatible endpoints (MinIO, Backblaze B2, Wasabi). The design must define how endpoints are validated (region vs endpoint URL precedence, path-style vs virtual-hosted addressing per provider), how TLS certificate validation is handled for self-hosted endpoints, how a region mismatch is detected (at configuration-time connection test versus first upload), and how the data-residency determination (Requirement 20.8/20.9) resolves an endpoint whose physical region is not declared.

21. **S3 access-key rotation.** Requirement 28.4 stores S3 credentials as operational secrets under `ENCRYPTION_MASTER_KEY` (distinct from the Backup_Data_Key rotation of Requirement 16.10). The design must decide how S3 access-key rotation is handled — re-entry of new keys, whether an overlap window is supported so in-flight uploads are not interrupted, whether the Requirement 28.7 connection test is automatically re-run on rotation, and how a rotation that invalidates the stored keys is surfaced (connection set to `disconnected` with a failure notification, consistent with the OAuth refresh-token revocation handling in Requirement 2.6).
